# §3. Features — 기능 요구사항

> PromptCraft 2.5의 기능 명세 본문. 35개 우선순위 매트릭스와 핵심 기능 절을 다루며, 시나리오2 실측 평가(2026-05-22)에서 입증된 항목을 명시적으로 표기하고, 디자인 시스템 통합·신규 기능 요구사항을 명세한다. UI 시각 명세는 [§4 UI-Design](./4.UI-Design.md), 평가 데이터 인용은 [§6 Evaluation](./6.Evaluation.md)을 참조한다.

본 절의 표기 규칙:
- ✅ — 시나리오2(refactoring, R1·R2·R3 × L1/L2/L3 × A/B 18단위 + 재시도 1단위) 실측 검증 항목.
- 🆕 — PRD 2.5 신설 또는 갱신 항목.
- (Pn) — 우선순위. P0 = MVP 필수, P1 = MVP 직후 확장, P2 = 후속.

## 3.2 핵심 기능 상세

### 3.2.1 SectionCard 모델 — 25개 카드 ✅

**12개 카드 속성:** `id`, `label`, `required`, `active`, `order`, `inputType`, `value`, `template`, `hint`, `examples`, `options`, `scanSuggested`, `defaultValue`.

**inputType 5종:** `text`, `multiline`, `select`, `select-or-text`, `multiline-mention`.

**25개 카드 ID 일람:**

| 그룹 | 카드 ID |
|------|---------|
| 공통 (필수) | `role`, `goal` |
| 컨텍스트 | `stack-environment`, `current-situation`, `constraints` |
| 에러 진단 | `error-evidence`, `tried-methods`, `build-log`, `request-log`, `profiling-data`, `baseline-metric`, `expected-behavior` |
| 구현 | `impl-scope`, `target-code`, `tech-preference`, `modification-scope`, `acceptance-criteria` |
| 리뷰 | `review-code`, `review-focus`, `security-context`, `concern-area` |
| 학습 | `concept`, `skill-level`, `output-pref` |
| 출력/예시 | `output-format`, `example-io`, `related-code` |

**빌드 규칙:** `active === true && value.trim() !== ''`인 카드만 `order` 오름차순 조립. 빈 값 active 카드는 프리뷰에서 회색 표시(시각적 압박) — 최종 출력에는 미포함. `{{value}}` 치환은 injection 방지를 위해 single-pass.

**시나리오2 실측 검증:**
- L3-B 시나리오에서 R1·R2·R3 3개 모델 모두 25개 카드 구조를 응답·구현에 정확히 반영. 핵심 3목표 충족률 9/9 (3모델 × G2/G3/G4 = 9 단위) 만점.
- 항목 누락 0건. H7("출력 항목 누락 감소") 가설이 두 평가 사이클 모두에서 가장 강하게 입증되었다. 자세한 표는 [§6 Evaluation](./6.Evaluation.md) §6.2.

### 3.2.2 트리 기반 카드 풀 — 5개 트리 ✅

| 트리 ID | 레이블 | 기본 활성 카드 수 |
|---------|--------|----------------|
| `error-solving` | 에러 해결 | 5 |
| `feature-impl` | 기능 구현 | 4 |
| `code-review` | 코드 리뷰 | 4 |
| `concept-learn` | 개념 학습 | 5 |
| `refactoring` | 리팩토링 | 5 |

**메커니즘 2종:**
- `cardOverrides` — 트리 JSON에서 카드별 hint/examples/defaultValue/options를 재정의. 예: `code-review`의 `goal.defaultValue = "코드 리뷰"`, `output-format`에 "심각도 분류표", "라인별 인라인 코멘트" 등 4종 옵션.
- `roleSuffix` — 역할 후보 끝에 접미사 부착. `code-review`("코드 리뷰 전문가"), `refactoring`("리팩토링 전문가")만 사용.

**시나리오2 실측 검증:** `refactoring` 트리를 18단위 모두에서 사용. 18/18 `cargo check --workspace` PASS (exit 0, 0 errors). 트리 자체의 효과(타 트리와의 비교)는 본 라운드에서 측정 대상이 아님.

### 3.2.3 도메인 적응형 + 3계층 오버레이 — 11개 도메인 ✅

**11개 도메인:** `web-frontend`, `web-backend`, `data-ml`, `mobile`, `systems`, `devops`, `game`, `desktop`, `cli`, `embedded`, `general`. 분류 로직: `src/core/scanner/domain-classifier.ts`.

**3계층 병합** (`src/core/builder/domain-overlay.ts`):

```
cardDefs (base) → treeOverrides → domainOverrides (최우선)
```

도메인 오버레이는 `cardOverrides` + `cardRelevance: { cardId: 'high'|'medium'|'low' }` 선언. 카드풀은 관련성 순(high→medium→low)으로 재정렬.

**역할 후보 파이프라인:**

```
스캔 → 도메인 → domainRoles[domain][treeId]
  + frameworkRoles[detected] (1~2개)
  + languageRoles[detected] (1개)
  + tree.roleSuffix
  → 중복 제거 후 상위 5개 칩
```

**매핑 테이블 (`data/role-mappings.json`):**
- `domainRoles` — 11도메인 × 6키(5트리 + default)
- `languageRoles` — 13개 언어
- `frameworkRoles` — 약 50종

**시나리오2 실측 검증:** Rust workspace(obscura-cli + 6 crate)를 `systems` 도메인으로 정확히 분류. Rust 언어 역할 후보가 정상 노출됨. 데스크탑 에코시스템 감지는 본 라운드 시나리오와 무관하나 최근 커밋(`28880c2 feat(scanner): 가중치 기반 도메인 분류 및 데스크탑 에코시스템 감지 추가`)으로 도메인 분류 정확도 향상 완료.

### 3.2.4 워크스페이스 레이아웃 — Three-Column

**Three-Column Workbench**를 메인 워크스페이스로 채택한다 (현재 구현 기준 — `c705d25 feat(web): 3-컬럼 워크스페이스 및 라이트/다크 테마 도입` 커밋).

**현재 레이아웃:**

```
┌────────────────────────────────────────────────────────────────────┐
│ TopBar 44px (로고·breadcrumb·domain badge·rescan·ActionBar·테마) │
├──────────────┬─────────────────────────────────┬───────────────────┤
│ CardPool     │ Editor (flex)                   │ Preview (440px)   │
│ Sidebar      │ - H1 + active/empty 카운트       │ - "preview" + 토큰 │
│ 240px        │ - SectionCard 리스트 (filled)    │ - mono dense       │
│ - dashed     │ - DnD 정렬 + 빈 값 시각 표시      │ - 빈 카드 italic    │
│   border 칩  │                                 │   "(비어 있음)"   │
│ - 트리별     │                                 │                   │
│   아이콘     │                                 │                   │
└──────────────┴─────────────────────────────────┴───────────────────┘
```

좌측 240px CardPoolSidebar(`w-60`), 중앙 flex editor(`px-6 py-6`, `max-w-3xl mx-auto`), 우측 440px preview aside(`bg-muted`). 시각 명세는 [§4 UI-Design](./4.UI-Design.md)에서 상세히 다룬다.

대안 레이아웃(Dual-Pane 등)의 사용자 선택형 옵션 도입은 후속 작업으로 검토한다 ([§7 Roadmap](./7.Roadmap.md) 참조).

**필수/비필수 카드:**
- 필수 카드(`required: true`)는 헤더에 `Lock` 아이콘 + shadcn `Badge`. 비활성화 불가.
- 비필수는 우측 상단 [×] 버튼으로 카드 풀 복귀.

### 3.2.5 @file 멘션 시스템 — 라인 범위 입증 ✅🆕

- **자동완성:** `GET /api/mention/suggest?root=&path=` — 최대 30개 후보, ↑↓ Enter Tab 탐색.
- **파일 읽기:** `POST /api/mention/read` — `@path:40-55` 라인 범위 지정.
- **공백 경로:** `@"path with spaces/file.ts"` 따옴표 지원 (`3565dce feat(mention): 공백 포함 경로 따옴표 멘션 지원`).
- **파일 미리보기:** mention 선택 시 인라인 15줄 + 줄번호 (`641839b feat(mention): 파일 라인 범위 멘션 및 미리보기 지원`).
- **보안:** `pathGuard.ts`로 `../` 차단 + `.env`/바이너리 차단.

**시나리오2 실측 검증 — 라인 범위 멘션의 정량 효과:**

L3 시나리오는 시나리오 본문 자체에 `@crates/obscura-cli/src/main.rs#L506-L735` 같은 라인 범위 핀포인트를 포함했고, 이 입력 형태가 모델 응답에 다음 효과를 만들었다(상세 인용: [§6 Evaluation](./6.Evaluation.md) §6.4):

| 효과 | 측정 |
|------|------|
| Critical 항목 누락 | L1 평균 1.0건 → L3 평균 0건 (R1·R2·R3 평균) |
| 응답-구현 위치 정합성 | L1 시나리오 변경 92% 응답 누락(R3 L1-A) → L3 응답에서 라인 범위 명시적 인용 |
| Constraints 위반율 | A군 0% / B군 0% (L3-B 6단위 전수) |
| 핵심 3목표 충족 | R1·R2·R3 × A·B = 6단위 모두 3/3 만점 |

라인 범위 멘션은 의도 모호성을 가장 효과적으로 차단하는 메커니즘이며, PromptCraft의 L3 효과 정량 입증의 핵심 매개체다.

`TreeSelect` 화면에서 프로젝트 경로 입력 시 **800ms debounce 후 자동 `POST /api/scan`** 트리거 (Pre-scan).

스캔 결과 활용:

### 3.2.8 Run as... 직접 실행 — fire-and-forget

**비용 제로 원칙 보존:** PromptCraft는 프롬프트 생성까지만, LLM 호출은 사용자 설치 외부 CLI가 수행. 서버에서 LLM API 토큰 소비 0.

**지원 Provider 4종:** `claude-code`, `gemini`, `copilot`, `codex`. 모두 cwd 지정 지원.

**API:** `POST /api/prompt/run` → `{ runId, pid, startedAt }`.

**파이프라인 6단계:**
1. runId UUID 생성
2. `.promptcraft/runs/<runId>.prompt.md` 기록
3. provider 화이트리스트 검증 + `which <binary>` 사전 확인
4. `Bun.spawn(binary, [...args, "--file", promptPath], { cwd, stdio: ["ignore", logFile, logFile], detached: true })`
5. `.promptcraft/runs/<runId>.log`에 stdout/stderr 파이프
6. 즉시 `{ runId, pid }` 반환 (결과 수신 없음)

**리스크 완화:** detach 후 PID만 기록 / 임시 파일 + `--file` 플래그 우선 + stdin 폴백 / provider 화이트리스트 고정 + 셸 경유 금지 / `which` 사전 확인 + 미설치 시 버튼 비활성화 / cwd는 스캔된 프로젝트 루트로 제한.

**히스토리:** `GET /api/runs/:runId/log`로 로그 조회.

**후속 보강(§3.3.1 참조):** 시나리오2에서 모델별 응답-구현 신뢰성 격차(R2 단언 과장, R3 scope 축소 보고)가 발견되어, Run as 완료 후 응답-구현 정합성 검증 단계를 신규 기능으로 추가한다. Run as 자체 사양은 변경되지 않는다.

### 3.2.9 DB 스키마 마이그레이션

`bun:sqlite` + `PRAGMA user_version` 기반. 현재 스키마 **v1**. `src/core/db/migrations/<version>.ts`의 `up(db)` 함수. 기동 시 자동. 실패 시 ① `~/.promptcraft/db.sqlite.bak.<ts>` 백업, ② 서버 중단, ③ stderr 출력. 단일 트랜잭션 + DDL 실패 시 전체 롤백. 테이블: `history`, `template`, `config`.

| `/api/templates` | GET, POST, DELETE | ⚠️ POST만 사용, GET/DELETE는 UI 없음 |
| `/api/config` | GET, PUT | ⚠️ Settings UI 미구현 |
| `/api/mention/suggest` | GET | ✅ MentionInput |
| `/api/mention/read` | POST | ✅ MentionInput (자동완성 + 파일 미리보기 양용) |

⚠️ 표기 항목은 API는 구현되었으나 UI 소비가 없는 상태다. PRD 2.5 P1 범위에서 History·Template 목록·Settings UI를 보강한다.

## 3.3 신규 기능 요구사항

시나리오2 평가에서 발견된 신규 패턴에 대응하는 신설 기능.

### 3.3.1 응답-구현 정합성 검증 (P1)

**배경:** 시나리오2 실측에서 모델별 응답-구현 신뢰성 격차가 신규 지표로 부상했다. AI가 응답 텍스트에 작성한 변경 내역과 실제 git 커밋의 차이를 정량화한 결과는 다음과 같다(상세: [§6 Evaluation](./6.Evaluation.md) §6.5).

| 모델 | 응답-구현 불일치 패턴 |
|------|----------------------|
| R1 | ±1 오차 수준의 정직 자가 보고 ("1337 → 1099"은 실측 1338 → 1100). 18단위 중 최상 |
| R2 | L2-B "타입 안전성 확보" 단언 vs 실제 enum 0건. L2-B "170줄"(실측 236줄, 1.4배 오차). 의도 충족도 **과장** 패턴 |
| R3 | L1-A에서 변경 92%(36/39 파일) 응답 누락. L1-A "main.rs 720줄"(실측 1117줄, 1.5배 오차). 변경 범위 **축소** 보고 패턴 |

PromptCraft 구조화는 B군에서 격차를 감소시켰으나 완전 해소하지 못했다. **응답 신뢰성은 사용자가 산출물을 적용하기 전 반드시 확인해야 하는 영역**이며, 이를 시스템 차원에서 보조해야 한다.

**기능 명세:**

`POST /api/prompt/run` 완료 후 Run as 로그(`/api/runs/:runId/log`)와 git 작업 트리의 변경(`git diff HEAD` 또는 직전 커밋)을 비교하는 후처리 단계 추가.

```
runId 완료 감지 → git diff 수집 → 응답 텍스트 파싱
  → 주장-변경 매핑 → 불일치 보고 카드 렌더
```

**검증 항목 (최소 셋):**

| 항목 | 검증 방법 | 출력 |
|------|----------|------|
| 변경 파일 수 정합성 | 응답의 "N개 파일 변경" 단언 vs `git diff --stat` 실측 | ±1 허용, 초과 시 경고 |
| 라인 수 정합성 | 응답의 "M줄"/"M줄에서 N줄로" vs `wc -l` 또는 diff 실측 | ±10% 허용, 초과 시 경고 |
| 변경 영역 정합성 | 응답에 언급된 파일/심볼 vs 실제 변경된 파일/심볼 집합 | 응답 누락 비율 출력, 50% 이상이면 critical 경고 |
| 단언 정합성 (선택) | "enum 도입", "타입 안전성", "분해" 등 핵심 단언이 실제 코드에 존재하는지 | LLM judge 또는 정적 분석 (P2 확장) |

**구현 옵션:**
1. **정적 분석 단독** — `git diff --stat` + 정규식으로 파일 수·라인 수만 비교. 외부 의존성 0, 구현 비용 최저.
2. **LLM judge 보조** — 응답 텍스트와 diff를 외부 LLM에 보내 단언-구현 매핑 평가. 비용 발생, 정확도 높음. **옵트인**으로 제공하여 비용 제로 원칙(§3.2.8) 보존.

PRD 2.5 P1 범위는 **옵션 1(정적 분석 단독)을 기본**으로 한다. LLM judge는 P2 확장으로 보류한다.

**UI:**
- 워크스페이스 우측 ActionBar 영역에 "응답-구현 검증" 토글 (기본 ON, 정적 분석 모드).
- Run as 완료 후 PromptPreview 하단에 "정합성 보고" 카드 노출 (불일치 있을 때만).
- 보고 카드는 dismissable이지만 ✗(미해결 경고) 상태에서는 다음 Run as까지 잔존.

**비기능 요구사항:**
- 정합성 검증은 Run as와 비동기적으로 실행. 사용자가 다음 작업으로 즉시 이동 가능.
- 검증 결과는 `~/.promptcraft/runs/<runId>.verify.json`에 영속화.
