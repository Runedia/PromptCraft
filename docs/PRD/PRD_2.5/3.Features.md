# §3. Features — 기능 요구사항

> PromptCraft 2.5의 기능 명세 본문. 31개 우선순위 매트릭스와 핵심 기능 절을 다루며, 시나리오2 실측 평가(2026-05-22)에서 입증된 항목을 명시적으로 표기하고, 디자인 시스템 통합·신규 기능 요구사항을 명세한다. UI 시각 명세는 [[4.UI-Design|§4 UI-Design]], 평가 데이터 인용은 [[6.Evaluation|§6 Evaluation]]을 참조한다.

본 절의 표기 규칙:
- ✅ — 시나리오2(refactoring, R1·R2·R3 × L1/L2/L3 × A/B 18단위 + 재시도 1단위) 실측 검증 항목.
- 🆕 — PRD 2.5 신설 또는 갱신 항목.
- (Pn) — 우선순위. P0 = MVP 필수, P1 = MVP 직후 확장, P2 = 후속.

## 3.2 핵심 기능 상세

### 3.2.1 SectionCard 모델 — 27개 카드 ✅

**13개 카드 속성:** `id`, `label`, `required`, `active`, `order`, `inputType`, `value`, `template`, `hint`, `examples`, `options`, `scanSuggested`, `defaultValue`. (정적 정의 `card-definitions.json`은 `label`·`required`·`inputType`·`template`·`hint`·`examples`·`scanSuggested`·`options` 등을 담고, `id`·`active`·`order`·`value`·`defaultValue`는 런타임·트리 오버레이에서 부여된다.)

**inputType 5종:** `text`, `multiline`, `select`, `select-or-text`, `multiline-mention`.

**27개 카드 ID 일람:**

| 그룹 | 카드 ID |
|------|---------|
| 공통 (필수) | `role`, `goal` |
| 컨텍스트 | `stack-environment`, `current-situation`, `constraints` |
| 에러 진단 | `error-evidence`, `tried-methods`, `build-log`, `request-log`, `profiling-data`, `baseline-metric`, `expected-behavior` |
| 구현 | `impl-scope`, `target-code`, `tech-preference`, `modification-scope`, `acceptance-criteria` |
| 리뷰 | `review-code`, `review-focus`, `security-context`, `concern-area` |
| 학습 | `concept`, `skill-level`, `output-pref` |
| 출력/예시 | `output-format`, `example-io`, `related-code` |

**빌드 규칙:** `active === true && value.trim() !== ''`인 카드만 `order` 오름차순 조립. 빈 값 active 카드는 프리뷰에서 회색 표시(시각적 압박) — 최종 출력에는 미포함. `{{value}}` 치환은 injection 방지를 위해 single-pass.

**시나리오2 실측 검증:**
- L3-B 시나리오에서 R1·R2·R3 3개 모델 모두 카드 구조를 응답·구현에 정확히 반영. 핵심 3목표 충족률 9/9 (3모델 × G2/G3/G4 = 9 단위) 만점.
- 항목 누락 0건. H7("출력 항목 누락 감소") 가설이 두 평가 사이클 모두에서 가장 강하게 입증되었다. 자세한 표는 [[6.Evaluation|§6 Evaluation]] §6.2.

### 3.2.2 트리 기반 카드 풀 — 5개 트리 ✅

| 트리 ID | 레이블 | 기본 활성 카드 수 |
|---------|--------|----------------|
| `error-solving` | 에러 해결 | 5 |
| `feature-impl` | 기능 구현 | 4 |
| `code-review` | 코드 리뷰 | 4 |
| `concept-learn` | 개념 학습 | 5 |
| `refactoring` | 리팩토링 | 5 |

**메커니즘 2종:**
- `cardOverrides` — 트리 JSON에서 카드별 hint/examples/defaultValue/options를 재정의. 예: `code-review`의 `goal.defaultValue = "코드 리뷰"`, `output-format`에 "심각도 분류표", "라인별 인라인 코멘트" 등 4종 옵션.
- `roleSuffix` — 역할 후보 끝에 접미사 부착. 5개 트리 전체 적용: `error-solving`("디버깅 전문가"), `feature-impl`("기능 구현 전문가"), `code-review`("코드 리뷰 전문가"), `refactoring`("리팩토링 전문가"), `concept-learn`("기술 멘토"). 4개 트리는 "전문가" 패턴 통일, 학습 맥락의 `concept-learn`만 "멘토"로 의도적 일탈.

**시나리오2 실측 검증:** `refactoring` 트리를 18단위 모두에서 사용. 18/18 `cargo check --workspace` PASS (exit 0, 0 errors). 트리 자체의 효과(타 트리와의 비교)는 본 라운드에서 측정 대상이 아님.

### 3.2.3 도메인 적응형 + 3계층 오버레이 — 11개 도메인 ✅

**11개 도메인:** `web-frontend`, `web-backend`, `data-ml`, `mobile`, `systems`, `devops`, `game`, `desktop`, `cli`, `embedded`, `general`. 분류 로직: `src/core/scanner/domain-classifier.ts`.

**3계층 병합** (`src/core/builder/domain-overlay.ts`):

```
cardDefs (base) → treeOverrides → domainOverrides (최우선)
```

도메인 오버레이는 `cardOverrides` + `cardRelevance: { cardId: 'high'|'medium'|'low' }` 선언. 카드풀은 관련성 순(high→medium→low)으로 재정렬.

> 도메인 분류와 역할 매핑은 11개 도메인 전부를 지원하나, 도메인 오버레이 파일(`data/domains/`)은 주요 도메인부터 점진 구현 중이다. 오버레이가 없는 도메인은 base → tree 단계로 폴백한다(미구현 오버레이 보강은 [[7.Roadmap|§7]] 과제).

**역할 후보 파이프라인 (5 슬롯 권장 분배):**

```
1   tree×조합 — `${Framework} ${tree.roleSuffix}`, `${Language} ${tree.roleSuffix}` (roleSuffix 있을 때)
2~3 base 역할 — domainRoles[domain].default 상위 2개  ← 모든 트리 공통, 통일성 핵심
4   framework 정제 역할 — frameworkRoles[detected] (1개)
5   tree-spec 역할 — domainRoles[domain][treeId] 상위 1개
─── 그 다음: domain default 잔여 → languageRoles(low conf 한정) → general fallback
중복 제거 후 호출자가 slice (server 5개, workspace 카드는 전체)
```

메인 화면(`/api/scan` 응답)은 `roleSuggestionsByTree: Record<treeId, string[]>` dict로 5개 트리 각각 5개 역할을 사전 계산하여 반환한다. 트리 선택 즉시 dict 조회만으로 워크스페이스의 `role` 카드 옵션과 1:1 일치 확보. 트리 미선택 상태에서는 `roleSuggestions`(default treeId 계산 결과) fallback 노출. (base prepend 알고리즘 재배치 및 `roleSuggestionsByTree` 도입 근거: [[2026-05-24-role-resolver-base-prepend|DECISIONS — base prepend]] · 5트리 `roleSuffix` 일관 적용: [[2026-05-24-trees-rolesuffix|DECISIONS — trees roleSuffix]].)

**매핑 테이블 (`data/role-mappings.json`):**
- `domainRoles` — 11도메인 × 6키(5트리 + default)
- `languageRoles` — 14개 언어
- `frameworkRoles` — 약 50종

**시나리오2 실측 검증:** Rust workspace(obscura-cli + 6 crate)를 `systems` 도메인으로 정확히 분류. Rust 언어 역할 후보가 정상 노출됨. 데스크탑 에코시스템 감지는 본 라운드 시나리오와 무관하나 최근 커밋(`28880c2 feat(scanner): 가중치 기반 도메인 분류 및 데스크탑 에코시스템 감지 추가`)으로 도메인 분류 정확도 향상 완료.

### 3.2.4 워크스페이스 레이아웃 — Three-Column

**Three-Column Workbench**를 메인 워크스페이스로 채택한다 (현재 구현 기준 — `c705d25 feat(web): 3-컬럼 워크스페이스 및 라이트/다크 테마 도입` 커밋).

**현재 레이아웃:**

```
┌────────────────────────────────────────────────────────────────────┐
│ TopBar 44px (로고·breadcrumb·domain badge·rescan·ActionBar·테마) │
├──────────────┬─────────────────────────────────┬───────────────────┤
│ CardPool     │ Editor (flex)                   │ Preview (440px)   │
│ Sidebar      │ - H1 + active/empty 카운트       │ - [원본] [미리보기] + 토큰 │
│ 240px        │ - SectionCard 리스트 (filled)    │ - mono dense / prose │
│ - dashed     │ - DnD 정렬 + 빈 값 시각 표시      │ - 빈 카드 italic    │
│   border 칩  │                                 │   "(비어 있음)"   │
│ - 트리별     │                                 │                   │
│   아이콘     │                                 │                   │
└──────────────┴─────────────────────────────────┴───────────────────┘
```

좌측 240px CardPoolSidebar(`w-60`), 중앙 flex editor(`px-6 py-6`, `max-w-3xl mx-auto`), 우측 440px preview aside(`bg-muted`). 시각 명세는 [[4.UI-Design|§4 UI-Design]]에서 상세히 다룬다.

**프리뷰 렌더 모드 토글.** 우측 패널 헤더 좌측에 `[원본] [미리보기]` underline tab 토글 2개(활성 시 하단 2px primary border). 원본(default)은 carbon-mono raw text로 LLM 입력과 1:1 시각 매핑을 보존하며, 미리보기는 `react-markdown` + `remark-gfm`으로 카드 본문을 prose 렌더하여 가독성을 제공한다. 토글 상태는 localStorage에 persist. 빌드 산출물(LLM에 전송되는 텍스트)은 양 모드 모두 동일하며 토글은 시각 표현만 변경한다. 상세 명세는 [[4.UI-Design#4.3.2 PromptPreview|§4.3.2]] / [[5.Architecture#5.4.4 `react-markdown` 도입 (미리보기 모드 전용)|§5.4.4]] 참조.

**필수/비필수 카드:**
- 필수 카드(`required: true`)는 헤더에 `Lock` 아이콘 + shadcn `Badge`. 비활성화 불가.
- 비필수는 우측 상단 [×] 버튼으로 카드 풀 복귀.

### 3.2.5 @file 멘션 시스템 — 라인 범위 입증 ✅🆕

- **자동완성:** `GET /api/mention/suggest?root=&partial=` — 최대 30개 후보, ↑↓ Enter Tab 탐색.
- **파일 읽기:** `POST /api/mention/read` — `@path:40-55` 라인 범위 지정.
- **공백 경로:** `@"path with spaces/file.ts"` 따옴표 지원 (`3565dce feat(mention): 공백 포함 경로 따옴표 멘션 지원`).
- **파일 미리보기:** mention 선택 시 인라인 15줄 + 줄번호 (`641839b feat(mention): 파일 라인 범위 멘션 및 미리보기 지원`).
- **보안:** `pathGuard.ts`로 `../` 차단 + `.env`/바이너리 차단.

**시나리오2 실측 검증 — 라인 범위 멘션의 정량 효과:**

L3 시나리오는 시나리오 본문 자체에 `@crates/obscura-cli/src/main.rs#L506-L735` 같은 라인 범위 핀포인트를 포함했고, 이 입력 형태가 모델 응답에 다음 효과를 만들었다(상세 인용: [[6.Evaluation|§6 Evaluation]] §6.4):

| 효과 | 측정 |
|------|------|
| Critical 항목 누락 | L1 평균 1.0건 → L3 평균 0건 (R1·R2·R3 평균) |
| 응답-구현 위치 정합성 | L1 시나리오 변경 92% 응답 누락(R3 L1-A) → L3 응답에서 라인 범위 명시적 인용 |
| Constraints 위반율 | A군 0% / B군 0% (L3-B 6단위 전수) |
| 핵심 3목표 충족 | R1·R2·R3 × A·B = 6단위 모두 3/3 만점 |

라인 범위 멘션은 의도 모호성을 가장 효과적으로 차단하는 메커니즘이며, PromptCraft의 L3 효과 정량 입증의 핵심 매개체다.

### 3.2.6 폴더 브라우저 모달

`GET /api/browse?path=` — Windows 드라이브 문자(`C:\`, `D:\`) 자동 감지, 상위 이동 + 직접 입력 지원. shadcn `Sheet`(우측 520px) 기반.

### 3.2.7 프로젝트 스캔 통합 + Pre-scan

`TreeSelect` 화면에서 프로젝트 경로 입력 시 **800ms debounce 후 자동 `POST /api/scan`** 트리거 (Pre-scan).

스캔 결과 활용:
- `stack-environment` 자동 채움
- 도메인 분류 → `role` 후보 생성
- `multiline-mention` 자동완성 루트 경로

**분석 항목:** 언어별 파일 수, 프레임워크(`package.json` 의존성, 40~50종), 디렉토리 구조(깊이 2~5), 패키지 매니저(bun/pnpm/yarn/npm), `.env` 존재, 설정 파일, gitignore 규칙, 도메인 분류.

### 3.2.8 Run as... 직접 실행 — 새 터미널 실행 + 클립보드

**비용 제로 원칙 보존:** PromptCraft는 프롬프트 생성까지만, LLM 호출은 사용자 설치 외부 CLI가 수행. 서버에서 LLM API 토큰 소비 0.

**지원 Provider 4종:** `claude-code`, `gemini`, `copilot`(`gh copilot`), `codex`. provider 레지스트리는 `src/core/run/providers.ts`(순수 데이터, 프론트·백엔드 공용).

**동작:** cwd(스캔된 프로젝트 루트)에서 **새 터미널 창을 열어 provider CLI를 실행**하고, 동시에 **프론트엔드에서 프롬프트를 클립보드에 복사**한다. 사용자는 새 창에서 `Ctrl+V`로 붙여넣어 사용한다. 하네스의 프롬프트 직접 실행(`--file`) 지원이 불확실하므로 프롬프트는 인자로 주입하지 않는다. 새 터미널은 spawn 시점에 레지스트리에서 읽은 최신 시스템+사용자 환경변수를 상속한다(PromptCraft 기동 시점 스냅샷이 아님). 수집 실패 시 앱 환경변수로 폴백한다.

**API:**
- `POST /api/prompt/run` → body `{ target, cwd }` → `{ ok, launched }`. 프롬프트는 전송하지 않는다(클립보드 전용).
- `GET /api/prompt/providers` → provider별 설치 여부 맵(`Bun.which`).

**셸 (확장 가능):** 새 창은 `cmd.exe /c start ""`로 띄우고 내부 셸이 CLI를 실행한다. 빌트인 `cmd`(기본)·`powershell`·`pwsh`. 셸은 DB config `run.shell`로 선택하고, `run.shells`(JSON)로 사용자 정의 셸(`{cwd}`/`{launch}` 토큰 템플릿)을 추가·오버라이드한다. `/api/config`로 변경. Settings UI는 후속.

**리스크 완화:** 프롬프트 인자 미주입(클립보드 전용) / provider 화이트리스트 고정 + `Bun.spawn(file, args[])` 셸 미경유 / `Bun.which` 사전 확인 + 미설치 provider 버튼 비활성화 / cwd는 스캔된 프로젝트 루트로 제한 + 존재·디렉토리 검증.

> **보류:** 완전한 fire-and-forget(`runId`·`.promptcraft/runs/*.prompt.md` 기록·`--file` 주입·stdout/stderr `.log` 파이프·`GET /api/runs/:runId/log`)은 하네스 인터페이스 불확실성으로 보류. 상세: [[2026-05-24-run-as-simplified]].

### 3.2.9 DB 스키마 마이그레이션 — 미구현 (P1-16)

> **미구현.** 본 절이 기술하는 `PRAGMA user_version` 기반 마이그레이션 프레임워크는 코드에 존재하지 않는다. 현재 `src/core/db/connection.ts`는 `CREATE TABLE IF NOT EXISTS`로 테이블 `history`·`config` 2종만 생성하며, 버전 관리·자동 적용·백업·롤백·기동 중단이 없다. 상세 trail: [[2026-05-24-feat29-migration-unimplemented]].

**목표 설계 (P1-16 구현 대상):** `bun:sqlite` + `PRAGMA user_version` 기반. `src/core/db/migrations/<version>.ts`의 `up(db)` 함수. 기동 시 자동 적용. 실패 시 ① `~/.promptcraft/db.sqlite.bak.<ts>` 백업, ② 서버 중단, ③ stderr 출력. 단일 트랜잭션 + DDL 실패 시 전체 롤백. 구현·우선순위는 [[7.Roadmap|§7.2.2]] P1-16 참조.

> **계획 취소.** 프롬프트 생성 외 접근(Run as 후처리, git diff 비교, LLM judge)은 프로젝트 범위를 벗어난다.
