# 2026-05-24 — §7 P1-3 재정의

**결정**: §7.2.2 P1-A의 P1-3을 "자동 제약 조건 매핑(`tsconfig`·`.eslintrc`·`Cargo.toml`에서 제약 추출 → `constraints` 카드 자동 채움, KPI #15 효과 확장)"에서 **"환경 제약 컨텍스트 보강"**으로 재정의한다. 새 정의는 tsconfig에서 *코드 생성 행동에 영향을 주는 소수 옵션만* 화이트리스트로 추출하여 자연어 행동 지침으로 정규화해 `stack-environment` 카드에 주입한다. eslint와 `constraints` 카드는 대상에서 제외한다. **번호(P1-3)와 우선순위(P1)는 유지**하며, 재번호화하지 않는다.

**근거**:
1. **기존 정당화의 수단-목적 불일치**: 구 P1-3은 "KPI #15 효과를 L3 시나리오 외 영역으로 확장"을 근거로 삼았으나, KPI #15가 효과를 입증한 제약(외부 crate 추가 금지·함수 시그니처 보존·파일 scope 한정, [[7.Roadmap|§7.1.2]])은 전부 **작업별 의도적 제약**이다. 설정 파일에서 추출 가능한 것은 컴파일러 옵션·린트 규칙·의존성 목록 같은 **환경 정적 속성**으로 종류가 다르다. KPI #15가 입증한 명제는 "사용자가 의도를 명시하면 B군이 지킨다"이지 "환경에서 추출한 제약이 효과적이다"가 아니다. 따라서 KPI #15와의 연결을 제거하고 정당화를 "모델에 환경 인지를 제공한다"로 정정한다.
2. **`constraints` 설계 역전 방지**: `constraints` 카드는 `scanSuggested: false`로 자동 채움을 의도적으로 배제하도록 정의되어 있다(`data/cards/card-definitions.json:61-71`). constraints는 본질적으로 *이번 작업의 사용자 의도*이지 프로젝트의 정적 속성이 아니므로, 환경 추출 값을 밀어넣으면 사용자가 강조하려는 제약을 흐리는 노이즈가 된다. 추출 정보는 이미 스캔 결과가 자동 주입되는 `stack-environment` 카드(`src/core/builder/cardSession.ts:35-37`)로 흡수한다.
3. **조합 폭발 통제 (화이트리스트 투영)**: tsconfig 컴파일러 옵션은 100개 이상이고 `extends` 상속·`references`로 경우의 수가 사실상 무한이다. 이를 일반 해석하면 폭발한다. 따라서 추출 대상을 *코드 생성 행동을 바꾸는 옵션*(`strict`/`strictNullChecks`/`noImplicitAny`, `noUncheckedIndexedAccess`, `module`/`moduleResolution`, `target`, `verbatimModuleSyntax`, `jsx` 등 ~6개)으로 **고정 화이트리스트화**하고, 나머지는 무시한다. 각 옵션은 자연어 행동 지침 한 줄로 *투영*한다(예: "strictNullChecks 활성 → null/undefined 명시 처리"). 입력 공간을 소수의 출력 공간으로 사영하여 폭발을 차단한다.
4. **eslint 제외**: flat config(`eslint.config.js`)는 JS 실행이 필요해 정적 파싱이 불가능하며, 임의 코드 실행은 PromptCraft의 셸·외부 실행 차단 보안 원칙과 충돌한다. legacy `.eslintrc`도 plugin·extends 조합 폭발. 현 스캐너도 `.eslintrc`·`tsconfig.json`을 존재 여부만 확인하고 내용은 파싱하지 않는다(`src/core/scanner/index.ts:13`). eslint는 범위에서 제외한다.
5. **`extends` 상속 위임**: tsconfig 상속 체인 머지는 직접 구현하지 않고 번들된 TypeScript의 config 파싱 API(`ts.parseJsonConfigFileContent` 계열)에 위임한다. MVP 단계에서는 최상위 `tsconfig.json` 명시 옵션만 읽어도 무방하다.
6. **P1 유지 사유**: 구 P1-3을 P2로 강등하는 대안을 검토했으나, 그 강등 근거는 "constraints 자동 채움 + 16개 에코시스템 전량 파싱"이라는 무거운 형태를 전제로 한 것이었다. 화이트리스트 투영으로 범위를 좁히고 대상을 tsconfig 단일 생태계 MVP로 한정하면 저비용·명확한 항목이 되어 P1 구현 가치가 회복된다. 따라서 강등 없이 P1 유지가 정합적이다.

**연쇄 정리**:
- [[7.Roadmap|§7.2.2]] P1-A 표 P1-3 행: 과제명 "자동 제약 조건 매핑" → "환경 제약 컨텍스트 보강", 설명을 화이트리스트 투영 방식으로 교체, 비고에 "stack-environment 확장" 명시.
- [[7.Roadmap|§7.3]] PRD 2.6 예상 영역 의존 컬럼의 P1-3 참조는 **그대로 유효**(번호·우선순위 불변). 변경 없음.
- [[7.Roadmap|§7.2.4]] 우선순위 매트릭스 P1-A `2건` **유지**. 변경 없음.
- `constraints` 카드의 `scanSuggested: false` 설계는 **유지**(본 재정의가 이를 명시적으로 존중). 카드 정의 변경 없음.
