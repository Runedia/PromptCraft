# §5 Architecture

PromptCraft의 시스템 아키텍처, 비기능 요구사항, 기술 스택, 디자인 시스템 정합성 결정 사항, 그리고 데이터 흐름·상태 관리를 정의합니다. 최근 커밋(`a654a1f` shadcn/ui 통합, `c705d25` 3-컬럼 워크스페이스 및 라이트/다크 테마)에서 확정된 구현 사실을 정확한 버전으로 반영합니다. UI 토큰·접근성 세부는 [[4.UI-Design|§4 UI-Design]]을, 기능 정의는 [[3.Features|§3 Features]]를 참조하십시오.

---

## 5.1 시스템 아키텍처

PromptCraft는 다섯 개의 계층으로 책임을 분리합니다. 각 계층은 자체 `CLAUDE.md`를 보유하며, 디렉터리 진입 시 컨벤션을 우선 확인합니다.

### 5.1.1 계층 도식

```
                        [사용자]
                            │  promptcraft serve
                            ▼
                ┌───────────────────────┐
                │ src/cli — serve 진입점 │  ── Commander, Bun 서버 기동
                └───────────────────────┘
                            │
                            ▼
                ┌───────────────────────┐
                │  src/web — React UI   │  ◀── Bun 서버가 정적 호스팅
                └───────────────────────┘
                            │  REST API (127.0.0.1)
                            ▼
                ┌───────────────────────┐
                │ src/server — Bun HTTP │  ── Run as: src/server/run/launcher가
                │   (+ run/launcher)    │     새 터미널 실행(shell: false, 클립보드 전용)
                └───────────────────────┘
                  │ (스캔·빌드)          │ (Run as)
                  ▼                      ▼
        ┌───────────────────────┐  ┌─────────────────────────────────────────┐
        │ src/core — 스캔·카드· │  │   외부 AI CLI 프로세스                  │
        │   빌더 (LLM-free)     │  │   (Claude Code / Gemini / Copilot /     │
        └───────────────────────┘  │    Codex CLI — 화이트리스트 4종)        │
                                   └─────────────────────────────────────────┘
```

`src/shared`는 모든 계층이 참조하는 타입 정의·상수·유틸리티의 공유 영역으로, 위 흐름의 측면에 위치하며 순환 의존을 발생시키지 않도록 설계합니다.

### 5.1.2 계층별 책임

| 계층 | 책임 | 외부 의존 | 컨벤션 |
|---|---|---|---|
| `src/web` | React UI(3-컬럼 워크스페이스), 상태 관리, 사용자 입력 수집, 프리뷰 렌더링 | Bun 서버의 REST API만 호출, 외부 LLM API 호출 0회 | [src/web/CLAUDE.md](../../src/web/CLAUDE.md) |
| `src/server` | HTTP 라우팅(12개 엔드포인트), 정적 자산 호스팅, CORS·CSP 헤더, Run as 화이트리스트 검증 + provider CLI 새 터미널 실행(`src/server/run/launcher`, `shell: false`) | `127.0.0.1` 바인딩 전용, 외부 네트워크 차단 | [src/server/CLAUDE.md](../../src/server/CLAUDE.md) |
| `src/core` | 프로젝트 스캔(다수 에코시스템 감지), 도메인 분류, 카드 조립, 프롬프트 빌더(Handlebars) | LLM API 호출 0회, 모든 처리 로컬 | [src/core/CLAUDE.md](../../src/core/CLAUDE.md) |
| `src/cli` | `promptcraft serve` 진입점(Commander), Bun 서버 기동 | Commander.js | [src/cli/CLAUDE.md](../../src/cli/CLAUDE.md) |
| `src/shared` | 타입 정의(Card·Scan 결과·Run 메타), 상수, 유틸리티 | 순환 의존 금지 | [src/shared/CLAUDE.md](../../src/shared/CLAUDE.md) |

### 5.1.3 의존 방향 원칙

- `web` → `server` (REST), `web` → `shared` (타입), `web` → `core` (Vite alias `@core`).
- `server` → `core`, `server` → `shared`.
- `cli` → `core`, `cli` → `server`(기동 시), `cli` → `shared`.
- `core` → `shared`.
- 역방향 의존 금지(`shared` → 다른 계층, `core` → `server`/`web`/`cli`).

## 5.2 비기능 요구사항

비기능 요구사항은 20개 항목으로 구성되며, 측정 조건이 미확정된 항목은 명시적으로 표기합니다.

### 5.2.1 실행 환경 / 네트워크 / 설정

| 항목 | 요구사항 |
|---|---|
| 실행 환경 | Windows / macOS / Linux. **Bun 1.3.10+ 필수, Node.js 비지원** |
| 네트워크 격리 | `127.0.0.1` 바인딩 전용, 외부 네트워크 차단. 프롬프트 생성 단계의 외부 LLM API 호출 0회 |
| 포트 충돌 | 기본 3000번, 사용 중 시 자동 다음 포트 탐색(최대 10회) |
| 설정 보존 | `~/.promptcraft/` 글로벌 또는 프로젝트 범위(`.promptcraft/`) |
| LLM 의존성 | 코어(스캔·카드 조립·빌드)는 LLM API 없이 100% 로컬 |

### 5.2.2 보안

| 항목 | 요구사항 |
|---|---|
| 보안 헤더 | CSP, X-Content-Type-Options, X-Frame-Options 강제 |
| CORS | `localhost` origin만 허용 |
| 경로 보안 | `../` 시도 403 차단, `.env` 및 바이너리 파일 차단 |
| Run as 보안 | Provider 화이트리스트(4종 고정), 셸 경유 금지(`shell: false`), `cwd`는 스캔된 프로젝트 루트로 제한 |
| 프로세스 실행 | `child_process.spawn`/`execFile`만 허용, `exec`/`execSync` 금지, `shell: true` 금지 |

| 빌드 성능 | 프롬프트 조립 5초 이내 | `POST /api/prompt/build` 응답 시간. 27개 카드 정의 중 25개 활성·예제 값 채움, 10회 측정, 중앙값 기준. **러너**: `tests/perf/api-build-benchmark.ts` (supertest). **npm**: `benchmark:api-build` |
| Undo/Redo | 카드 조작 **10단계**(zundo temporal middleware) | — |

Run as detach 프로세스는 영향 없음 | `child_process.spawn`으로 서버를 자식 프로세스로 띄운 뒤 시그널 발송 → 종료 시간·exit code 측정. **러너**: `tests/perf/graceful-shutdown-check.ts`. **npm**: `benchmark:shutdown`. **주의**: Windows에서는 `process.kill`이 graceful 핸들러를 우회하므로 측정 skip; POSIX 환경에서 실측 |

### 5.2.4 신뢰성 / 사용자 안전

| 항목 | 요구사항 |
|---|---|
| 단계별 안전성 | 스캔 → 카드 조립 → 프롬프트 생성 → Run as 각 단계가 독립 실패 가능. 한 단계 실패가 이전 단계의 데이터를 손상시키지 않음 |
| Undo/Redo 범위 | 카드 추가·삭제·재배치·내용 편집 10단계, 메모리 기반 |
| 오류 메시지 | 사용자 행동 기반(예: "프로젝트 경로가 비어 있습니다"), 시스템 메시지 직접 노출 금지 |

### 5.2.5 접근성

UI 토큰·키보드 내비게이션·`focus-visible`·`prefers-reduced-motion` 등 접근성 세부 사양은 [[4.UI-Design|§4 UI-Design]]에 정의됩니다. 본 절에서는 비기능 요구사항으로서 다음을 강제합니다.

| 항목 | 요구사항 |
|---|---|
| 키보드 내비게이션 | 모든 카드 조작 가능, focus trap 정확성, Tab order 명시 |
| 대비비 | WCAG AA 4.5:1 이상(본문), AAA 7:1 목표(중요 정보) |
| 모션 감소 | `prefers-reduced-motion: reduce` 사용자에게 애니메이션 제거 또는 단축 |
| 스크린 리더 | 카드 컴포넌트 ARIA 레이블, 드래그앤드롭 라이브 영역 알림 |

### 5.2.6 품질 / 테스트

| 항목 | 요구사항 |
|---|---|
| 타입 안정성 | TypeScript strict, 컴파일 에러 Zero |
| 테스트 커버리지 | 라인 커버리지 전역 **90% 이상**(`bun test --coverage`) |
| E2E 테스트 | Playwright 기반 |
| API 테스트 | supertest 기반 12개 라우트 전체 |
| 린트/포맷 | Biome 강제, `biome.json` 기준(single quotes, 2-space indent, LF, lineWidth 160, trailingCommas es5) |
| 커밋 검증 | Husky + commitlint(Conventional Commits) |

### 5.2.7 반응형 레이아웃

반응형 요구사항은 다음 항목으로 구성됩니다.

| 브레이크포인트 | 레이아웃 |
|---|---|
| ≥ 1024px | 3-컬럼 워크스페이스(좌측 카드 풀 / 중앙 작업 영역 / 우측 프리뷰), 기본 데스크탑 |
| 768–1023px | 단일 패널 + 프리뷰 토글(탭 전환) |
| < 768px | 단일 패널 + floating 프리뷰 시트(모바일, P2) |

## 5.4 디자인 시스템 정합성 결정 사항

본 절은 현재 디자인 시스템의 후속 결정 사항을 정리합니다.

### 5.4.1 폰트 로딩 방식

| 구분 | 변경 전 | 변경 후 |
|---|---|---|
| 로딩 방식 | `@import url("https://fonts.googleapis.com/css2?...")` (Google Fonts CDN) | `@fontsource/*` 패키지(Geist·Geist Mono·Instrument Serif) + GitHub raw 직접 호스팅(D2Coding) 셀프 호스팅 |
| 출처 | `src/web/styles/design-system.css:1` (제거됨) | `src/web/main.tsx`에서 fontsource weight별 CSS import (Geist 400/500/600/700, Geist Mono 400/500, Instrument Serif 400/400-italic) + `design-system.css` 상단 `@font-face`로 D2Coding 400/700 직접 정의 (`src/web/public/fonts/d2coding/D2Coding-Regular.ttf`, `D2Coding-Bold.ttf`) |
| CSP | `style-src` 에 `https://fonts.googleapis.com`, `font-src` 에 `https://fonts.gstatic.com` 화이트리스트 | 외부 호스트 제거, `style-src 'self' 'unsafe-inline'; font-src 'self';` 로 축소 (`src/server/middleware/security.ts`) |
| PRD 정합성 | §5.2.1 네트워크 격리 원칙과 충돌 | 외부 CDN 의존 0건, 네트워크 격리 강화 |
| 상태 | — | **완료**(P1, 2026-05-23) |

**결정**: 셀프 호스팅으로 전환 완료. fontsource 패키지는 `url(./files/...)` 참조를 Vite가 자동 번들링하여 hash 처리된 정적 자산으로 출력하며, D2Coding TTF는 `src/web/public/fonts/d2coding/`에 직접 보관하여 빌드 시 `dist/web/fonts/d2coding/`로 그대로 복사된다. D2Coding은 `unicode-range`(Hangul 범위 U+AC00-D7AF 등)로 한글 글리프 한정 적용되며, 영문 글리프는 chain 앞쪽의 Geist Mono로 처리되어 mono 시각 컨셉을 유지한다. 외부 CDN 호출은 0건.

### 5.4.2 토큰 네이밍

| 항목 | 상태 |
|---|---|
| 현재 구현 | shadcn 표준 토큰(`--background`, `--foreground`, `--primary` 등) + 비shadcn 보조 토큰(`font-code`, `font-ui`, `shadow-card`, `shadow-card-hover`, `shadow-drag`) |
| 도입 시점 | 커밋 `a654a1f shadcn/ui 통합 및 디자인 토큰 표준화` |
| 보강 검토 토큰 | `--pc-bg-inset`, `--pc-bg-overlay`, `--pc-border-inset`, `--pc-fg-subtle`, `--pc-accent-hover`, `--pc-accent-border`, `--pc-info`, `--pc-radius-xl` (현 화면 미사용이나 후속 화면 도입 시 동시 추가) |
| 우선순위 | **P1**(`@theme` 블록 보강) |

**결정**: shadcn 표준 토큰 네이밍을 유지하되, amber accent 계열(`--pc-accent-hover`, `--pc-accent-border`) 및 `--pc-info`는 `@theme` 블록에 보강합니다(P1).

### 5.4.3 테마 시스템

### 5.4.4 `react-markdown` 도입 (미리보기 모드 전용)

| 항목 | 상태 |
|---|---|
| 의존성 | `react-markdown ^9.1.0`(`package.json`) + `remark-gfm`(신규 추가) |

**결정**: `PromptPreview`에 `원문 / 미리보기` 토글을 도입하고, 미리보기 모드에서만 `react-markdown` + `remark-gfm`으로 카드 본문을 prose 렌더합니다. 원문 모드는 default이며 carbon-mono raw text를 그대로 유지하여 "LLM이 받는 텍스트와 1:1 시각 매핑" 컨셉을 보존합니다. 토글 상태는 localStorage에 persist합니다(zundo temporal 제외). 헤더 토글 명세는 [[4.UI-Design#4.3.2 PromptPreview|§4.3.2]] 참조.

## 5.5 데이터 흐름 / 상태 관리

### 5.5.1 상태 관리 토폴로지

PromptCraft Web의 상태는 Zustand 스토어 두 개로 분리됩니다.

| 스토어 | 역할 | 미들웨어 |
|---|---|---|
| `useCardStore` | 카드 컬렉션·배치·내용 편집·드래그앤드롭 결과 | `zundo` temporal middleware |
| `useTemporalStore` | undo/redo 히스토리 노출 | — |

`zundo`의 `partialize` 옵션을 통해 카드 콘텐츠·순서만 히스토리에 포함하고, UI 상태(드래그 중 위치, 호버 상태 등)는 제외합니다. 히스토리 한도는 **10단계**(§5.2.3).

### 5.5.2 데이터 흐름

```
[사용자 입력]
    │
    ▼
[useCardStore 액션] ── debounce 800ms ──▶ [POST /api/scan]
    │                                          │
    │                                          ▼
    │                                    [src/core: ScanEngine]
    │                                          │
    │  scan result(domain, frameworks)         │
    │  ◀──────────────────────────────────────┘
    │
    ▼
[useCardStore 갱신: stack-environment 자동 채움]
    │
    ▼
[PromptPreview 컴포넌트] ── 즉시 갱신 (< 100ms 목표) ──▶ 사용자
    │  (프롬프트는 클라이언트 @core/builder로 로컬 빌드 — 비용 제로)
    │
    ▼ [Run as] 버튼 클릭
    │
[POST /api/prompt/run] body { target, cwd }
    │
    ▼
[src/server: launcher] ── 새 터미널 창에서 provider CLI 실행
    │                       Bun.spawn(file, args[], { shell: false }) + 최신 시스템 env 주입
    │                       + 프론트엔드는 프롬프트를 클립보드에 복사
    ▼
[외부 AI CLI 프로세스] ── 사용자가 새 창에서 Ctrl+V로 붙여넣어 사용
```

### 5.5.3 debounce / 호출 타이밍

| 이벤트 | debounce | 트리거 API |
|---|---|---|
| 프로젝트 경로 입력 → Pre-scan | 800ms | `POST /api/scan` |
| 카드 내용 편집 → 프리뷰 갱신 | 0ms(즉시) | 클라이언트 사이드 렌더 |
| `@mention` 자동완성 | 200ms | `GET /api/mention/suggest` |
| Run as 버튼 | 0ms(즉시) | `POST /api/prompt/run` (프롬프트 빌드는 클라이언트 로컬) |

### 5.5.4 영속화

| 데이터 | 저장 위치 | 수명 |
|---|---|---|
| 프롬프트 히스토리 | `~/.promptcraft/db.sqlite`(`bun:sqlite` `history` 테이블) | 영구(사용자가 명시 삭제 시까지) |
| 사용자 설정 | `~/.promptcraft/db.sqlite`(`config` 테이블) | 영구 |

> Run as 단순화([[2026-05-24-run-as-simplified]])로 `runId`·생성 프롬프트 파일(`.prompt.md`)·실행 로그(`.log`)·`runs.sqlite`는 생성하지 않는다. 프롬프트는 클립보드로만 전달된다.
