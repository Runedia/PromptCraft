# §4. UI Design

> PromptCraft 2.5 UI 디자인 명세. 본 장은 **현재 구현(`src/web/`)을 SSOT**로 삼아 디자인 토큰, 컴포넌트, 화면, 인터랙션, 거버넌스를 명문화한다. 디자인 결정의 산물은 코드에 직접 반영되며, 별도의 디자인 산출물(Figma·외부 mock 등)을 거치지 않는다.

관련 문서: [[1.Overview|§1 Overview]] · [[3.Features|§3 Features]] · [[5.Architecture|§5 Architecture]] · [[6.Evaluation|§6 Evaluation]] · [[7.Roadmap|§7 Roadmap]]

## 4.1 디자인 원칙

### 4.1.1 미감 — v0/Vercel 계열 warm-neutral

PromptCraft 2.5의 시각 언어는 v0/Vercel 계열의 절제된 모노크롬 미감을 따른다. 핵심 특징은 다음과 같다.

- **Warm-neutral monochrome**: 순 회색 대신 미세하게 따뜻한 무채색을 사용한다. 라이트 모드 기준 base는 `#ffffff`, 텍스트는 `#0a0a0a`, border는 `#e7e5e2`, muted 배경은 `#fafaf9`.
- **단일 절제된 액센트**: warm amber/ember 계열(`#c2410c` 라이트 / `#fb923c` 다크) 1색만 액센트로 사용한다. 도메인 라벨, 마크다운 헤더, 트리 선택 표시, primary 버튼 등 의미상 강조가 필요한 지점에만 적용한다.
- **타이포그래피 위주 시각 위계**: 그라데이션·이미지·장식 도형을 배제한다. 위계는 폰트(Geist sans / Geist Mono / Instrument Serif)와 mono uppercase 캡션, 정확한 픽셀 크기·간격으로만 표현한다.
- **외부 이미지 자산 없음**: 모든 아이콘은 `lucide-react` import로 단일화한다. 정적 이미지·일러스트 자산은 0개를 유지한다.

### 4.1.2 라이트/다크 페어 (수동 토글)

모든 색상 토큰은 라이트·다크 **쌍(pair)** 으로 정의된다. 다크 모드는 `<html class="dark">` 클래스 토글로 활성된다. `next-themes`의 `attribute="class"` 모드를 사용하며, 첫 진입 시 OS의 `prefers-color-scheme`을 따른다 (`enableSystem`, `defaultTheme="system"`, `src/web/components/ThemeProvider.tsx:6`). 사용자가 토글을 누르면 명시 선택이 localStorage에 저장되어 시스템 변경과 무관하게 유지된다.

### 4.1.3 폰트

다음 3종을 사용한다.

| 패밀리 | 역할 | 라이선스 | 요청 weight |
| --- | --- | --- | --- |
| Geist | UI body (sans) | SIL OFL 1.1 | 400 / 500 / 600 / 700 |
| Geist Mono | 코드·경로·kbd·preview·mono caption (영문 글리프) | SIL OFL 1.1 | 400 / 500 |
| D2Coding | 한글 mono caption·preview body·카드 라벨 (한글 글리프 한정, `unicode-range` 매핑) | SIL OFL 1.1 | 400 / 700 |
| Instrument Serif | Serif H1 후보 (현재 사용처 0건, 토큰만 정의) | SIL OFL 1.1 | regular / italic |

로딩 방식은 `@fontsource/*` 패키지(Geist·Geist Mono·Instrument Serif)와 GitHub raw 직접 호스팅(D2Coding TTF, `src/web/public/fonts/d2coding/`)을 병용한 셀프 호스팅이다. weight별 CSS를 `src/web/main.tsx`에서 import하고 D2Coding은 `design-system.css`의 `@font-face`에서 직접 정의하며, Vite가 정적 자산을 자동 번들링하여 외부 CDN 호출이 발생하지 않는다. 상세는 [[5.Architecture|§5.4.1]] 참조.

### 4.1.4 접근성 원칙

| 원칙 | 적용 방식 |
| --- | --- |
| `focus-visible` | shadcn(Radix) 컴포넌트는 자체 focus-visible 제공. 비shadcn 영역(SectionCard 본체, TreeGrid 타일)도 `focus-within:ring-2 focus-within:ring-ring` 적용 (`src/web/components/SectionCard/SectionCard.tsx:46`) |
| `aria-label` | 모든 아이콘 전용 버튼에 부착 (`ChevronLeft`/`X`/`Sun`·`Moon` 등) |
| Landmark roles | `<header>`, `<main>`, `<aside>`, `<nav>` 의미 태그 사용 |
| Hydration-safe | `ThemeToggle`에 `mounted` 가드 적용 (`src/web/components/ThemeToggle.tsx:9-13`) |
| `prefers-reduced-motion` | 후속 개선 항목 (§4.6) |
| Skip-link | 후속 개선 항목 (§4.6) |

### 4.1.5 디자인 시스템 SSOT

토큰의 단일 출처는 다음 한 파일이다.

```
src/web/styles/design-system.css   (Tailwind v4 @theme 블록)
```

별도 CSS 파일 신규 생성은 금지된다 (`src/web/CLAUDE.md:7`). 모든 스타일은 `@theme` 토큰 + Tailwind 유틸리티 클래스 + 비shadcn 보조 토큰(`font-code`, `font-ui`, `shadow-card`, `shadow-card-hover`, `shadow-drag`) 조합으로만 작성한다.

## 4.3 Primitives — 공통 컴포넌트

본 절은 `src/web/components/` 하위 컴포넌트의 명세를 정리한다.

### 4.3.1 SectionCard

**위치**: `src/web/components/SectionCard/SectionCard.tsx`, `CardInput.tsx`

| Prop | 타입 | 기본값 | 설명 |
| --- | --- | --- | --- |
| `card` | `SectionCardType` | — | 카드 데이터 (id, label, value, required, inputType, hint, examples, options) |
| `variant` | `'outlined' \| 'filled'` | `'outlined'` | outlined = `bg-card border` + `shadow-card`, filled = `bg-muted/60 border-transparent` |
| `scanRoot` | `string?` | — | mention 처리 시 base path |

**상태별 시각 표현**:

| 상태 | 효과 |
| --- | --- |
| Idle (outlined) | `bg-card border border-border/50 shadow-[var(--shadow-card)]` |
| Idle (filled) | `bg-muted/60 border border-transparent` |
| Focus-within | `ring-2 ring-ring border-ring` |
| Dragging | `bg-accent shadow-[var(--shadow-drag)] border-primary ring-2 ring-primary/30 opacity-85` |
| Required | shadcn `Badge variant="secondary"` + `Lock` 아이콘 + "필수" 텍스트 |

**구조**: 상단 행 = grip 핸들(`GripVertical 16px`) + label(`text-lg font-semibold`) + 필수 배지 또는 제거 버튼(`X 16px`, hover 시 `text-destructive bg-destructive/10`). 하단 = `<CardInput>`이 `card.inputType`에 따라 `TextInput` / `MultilineInput` / `MentionInput` / `SelectOrTextInput` / shadcn `Select` 분기 렌더.

**DnD**: `@dnd-kit/sortable` `useSortable` 훅으로 카드 단위 정렬. `attributes`/`listeners`는 grip 버튼에만 부착되어 텍스트 입력 영역의 키보드를 방해하지 않는다 (`SectionCard.tsx:25-34, 55-56`).

### 4.3.2 PromptPreview

**위치**: `src/web/components/PromptPreview/PromptPreview.tsx`

워크스페이스 우측 440px 패널에 마운트된다. 헤더의 `[원본] / [미리보기]` underline tab 토글로 두 렌더 모드를 전환한다. 원본(default)은 carbon-mono raw text 렌더러를, 미리보기는 `react-markdown` + `remark-gfm`을 사용한다([[5.Architecture#5.4.4 `react-markdown` 도입 (미리보기 모드 전용)|§5.4.4]] 참조). 토글 상태는 localStorage persist.

**레이아웃**:

```
┌─ Header (h-9, border-b)
│  ├ 좌: [원본]   [미리보기]   (font-code 11px uppercase tracking-[0.07em], gap-5)
│  │     · 활성 버튼  : border-b-2 border-primary + text-foreground
│  │                    (-mb-px로 헤더 border-b 위에 오버레이)
│  │     · 비활성 버튼: border-b-2 border-transparent + text-muted-foreground,
│  │                    hover:text-foreground
│  └ 우: "~{N} tokens" (font-code 11px muted)
├─ Body (overflow-y-auto, px-6 py-5)
│  ├ [원문 모드] font-code 12px leading-1.7
│  │  ├ Filled cards :  "## {label}" (primary 12.5px 600)
│  │  │                  {value}      (whitespace-pre-wrap, secondary-foreground)
│  │  ├ Empty cards  :  opacity-40
│  │  │                  "## {label}" (muted-foreground)
│  │  │                  "(비어 있음 — 출력에서 제외)" (italic font-sans muted)
│  ├ [미리보기 모드] className="prose-preview" (design-system.css:155-187)
│  │  ├ Filled cards :  ReactMarkdown(`## {label}\n\n{value}`) + remarkGfm
│  │  │                  표·체크박스·자동링크 GFM 활성
│  │  ├ Empty cards  :  원문 모드와 동일 처리(markdown 외부 회색 헤더 + italic 안내)
│  └ Footer       :  border-t border-dashed
│                    "{N} sections active" / "~{N} tokens"
└─ Blank state (active 카드 0개) : 중앙 정렬 "카드를 채우면 프롬프트가 표시됩니다."
```

**상태 소스**: `useCardStore`의 `cards`·`tokenEstimate` + `useUIStore`의 `previewMode: 'raw' | 'rendered'`(default `'raw'`, localStorage `promptcraft-preview-mode` key persist). active+value 카드와 active+empty 카드를 두 모드 모두에서 시각적으로 구분한다.

### 4.3.4 ActionBar

**위치**: `src/web/components/ActionBar/ActionBar.tsx`

워크스페이스 TopBar 우측에 마운트된다. `forwardRef<ActionBarHandle>`로 외부 단축키 연동용 imperative API를 노출한다.

| Action | 시각 | 단축키 | 동작 |
| --- | --- | --- | --- |
| Undo | `Undo2 15px` ghost icon, size-8 | `⌘Z` | `useTemporalStore.undo()` |
| Redo | `Redo2 15px` ghost icon, size-8 | `⌘⇧Z` / `⌘Y` | `useTemporalStore.redo()` |
| Copy | `Copy 14px` outline button, h-8 + kbd hint `⌘↵` | `⌘↵` | `navigator.clipboard.writeText(prompt)` + sonner success |

### 4.5.3 Mention 흐름

1. textarea에서 `@` 입력 → 정규식 매칭 → `mentionStart` 기록
2. 150ms 디바운스 후 `GET /api/mention/suggest?root=...&partial=...` 호출 → suggestions 표시
3. 사용자가 항목 hover/선택 → 200ms 디바운스 후 `POST /api/mention/read`로 파일 미리보기 로드 (file일 때만)
4. `Enter`: 항목 삽입 (dir이면 자동 expand하여 하위 목록 재요청, file이면 close)
5. `⇧↵`: `@path#L|` 형식으로 삽입 + 커서를 `L|` 위치로 이동
6. `Esc` 또는 매칭 실패: popover close

### 4.5.4 Run dropdown 흐름

1. 사용자가 ActionBar의 "Run as" 버튼 클릭 → DropdownMenu 4 옵션 표시
2. 옵션 선택 → `POST /api/prompt/run`에 `{ target, cwd }` 전송(프롬프트는 미전송 — 클립보드 전용). 동시에 프론트엔드가 프롬프트를 클립보드에 복사
3. 응답 `{ ok, launched }` → cwd에서 새 터미널 창이 열려 provider CLI 실행(최신 시스템 환경변수 상속). sonner toast로 실행 안내 표시. 사용자는 새 창에서 `Ctrl+V`로 붙여넣어 사용
4. 실패 시 `toast.error('실행 실패: ...')`
5. `⌘⇧↵` 단축키는 항상 default target(`claude-code`)을 호출

> 동작·API·셸 상세는 [[3.Features|§3.2.8]], 보류 trail은 [[2026-05-24-run-as-simplified]] 참조.

## 4.7 디자인 시스템 거버넌스

### 4.7.1 토큰 변경 절차

1. **단일 SSOT**: `src/web/styles/design-system.css`의 `@theme` 블록 + `:root`/`.dark` 블록만 수정한다. 다른 CSS 파일을 신규로 만들지 않는다 (`src/web/CLAUDE.md:7`).
2. **라이트/다크 페어 강제**: 새 색상 토큰을 추가할 경우 `:root`(라이트)와 `.dark` 모두에 동일 키를 추가한다.
3. **Tailwind v4 노출**: 새 토큰을 Tailwind 유틸로 노출하려면 `@theme inline` 블록(`design-system.css:112-140`)에 `--color-{name}: var(--{name})` 항목을 추가한다.
4. **테스트**: 두 모드에서 시각 검증. 라이트→다크 토글 시 깜빡임·대비 부족 없는지 확인.

### 4.7.2 컴포넌트 추가 절차

1. **shadcn 우선**: 신규 primitive가 필요하면 먼저 `bunx --bun shadcn@latest add <component>`로 shadcn 원본을 추가한다. `src/web/components/ui/`에 설치된다.
2. **shadcn 위에 wrap**: 도메인 특화 컴포넌트는 `src/web/components/[Feature]/`에 작성하고, 내부적으로 shadcn primitives를 조합한다.
3. **`data-ui-id` 부착**: 모든 주요 UI 요소에 `data-ui-id={UI_IDS.X}` 속성을 부착한다. ID는 `src/web/ui-ids.ts`에 JSDoc(`@screen`, `@region`, `@description`, `@file`)과 함께 등록한다.
4. **ui-map 재생성**: `bun run ui-map:generate` 실행 → `docs/ui-map.md` 갱신 (자동 생성, 수동 편집 금지).
5. **import alias 강제**: 같은 디렉토리는 `./`, 그 외 `src/web` 내부는 `@/...`, `src/core` 참조는 `@core/...` (`src/web/CLAUDE.md:13-18`). 상대경로 `../`은 금지.

### 4.7.3 디자인 변경 정책

- 디자인 변경은 **코드에서 직접 반영**한다. 별도의 디자인 산출물(외부 mock, Figma 등)을 SSOT로 두지 않는다.
- 변경 사항은 PR description에 시각 비교 스크린샷 또는 영상으로 첨부한다.
- 토큰 변경 시 영향 범위를 PR description에 명시한다 (`bg-background`, `text-foreground` 등 광범위 토큰은 특히).

### 4.7.4 코드 스타일 (biome)

`biome.json`을 준수한다: single quotes, 2-space indent, LF, lineWidth 160, trailingCommas `es5` (루트 `CLAUDE.md` 참조).
