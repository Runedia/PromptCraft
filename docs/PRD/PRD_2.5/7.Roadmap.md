# §7 Roadmap

PromptCraft의 성공 지표(KPI)와 향후 과제를 정의합니다. 시나리오2 평가(R1·R2·R3 × L1·L2·L3 × A·B = 18단위, [[6.Evaluation|§6 Evaluation]] 참조)에서 도출된 실측 데이터와 디자인-구현 격차([[4.UI-Design|§4 UI-Design]] 참조)를 반영하여 KPI를 정의하고 과제 우선순위를 배치합니다.

KPI는 시나리오2 결과로 일부 확정되며, 미측정 항목은 P0 측정 인프라 과제로 명시적으로 이관합니다. 향후 과제는 P0(다음 사이클 전제) / P1(핵심 기능 개선) / P2(후속 기능 확장)의 3단계 우선순위로 분류합니다.

## 7.1 성공 지표 (KPI)

### 7.1.1 자동 검증 / 벤치마크 KPI

전체 15개 지표 중 자동 테스트로 검증 가능한 PromptCraft 자체 동작 항목(#3~#12)과 A/B 벤치마크가 필요한 항목(#1~#2)을 정리합니다. 시나리오2 평가에서 실측이 확보된 항목에는 ✅ 마커를 표기하고, 측정 불가 항목은 P0 과제와 연결합니다.

| # | 지표 | 측정 방법 | 시나리오2 실측 | 비고 |
|---|---|---|---|---|
| 1 | One-Shot 응답 성공률 | A/B 벤치마크, 빌드 통과 + 핵심 목표 충족 | ✅ B군 100% (L3 18/18 PASS), A군 100% (G1 한정) | 시나리오 평가에서 측정 |
| 2 | 학습 곡선 — 카드 사용률 감소 | 종단 연구 (세션 누적) | (TBD) | 단발 측정 불가 |
| 3 | 섹션 제어 정확성 | active 카드만 프롬프트에 포함되는지 단위 테스트 | ✅ 자동 테스트 PASS | — |
| 4 | 빈 헤더 출력 zero | value가 비어있는 active 카드 미출력 자동 검증 | ✅ 자동 테스트 PASS | — |
| 5 | 프리뷰 반응속도 | 카드 입력 → DOM 갱신 100ms 이내 | ✅ [[3.Features|§3 Features]] 기준 충족 | — |
| 6 | LLM 호출 zero | 코어 플로우 내 외부 API 호출 부재 검증 | ✅ 자동 테스트 PASS | — |
| 7 | Run as 보안 | Provider 화이트리스트 우회 시도 차단, 셸 경유 실행 부재 | ✅ 통합 테스트 PASS | — |
| 8 | 구동 안정성 | TypeScript 컴파일 에러 Zero, 빌드 단계 에러 Zero | ✅ CI PASS | — |

| 12 | DB 마이그레이션 | 기동 시 자동 적용 성공, 실패 시 백업 후 중단 동작 검증 | (미구현 — P1-16 과제) | [[2026-05-24-feat29-migration-unimplemented]] |
**해석.** 후반부 지표 중 #3~#11(9개)은 자동 테스트로 검증 가능한 PromptCraft 자체 동작 항목이며 시나리오2 평가에서 회귀 없이 통과했습니다. #12(DB 마이그레이션)는 코드 미구현이며 P1-16 과제로 이관되었습니다([[2026-05-24-feat29-migration-unimplemented]]). 전반부 2개 지표 중 #1(One-Shot)은 부분 확정되었고 #2(학습 곡선)는 현재 프로젝트 범위에서 제외합니다.

### 7.1.2 평가 도출 KPI

시나리오2 평가에서 PromptCraft 효과가 가장 결정적으로 입증된 영역과 모델 신뢰성 격차가 드러난 영역을 정식 KPI로 등재합니다. 정의·측정 방법·목표의 3컬럼 형식을 채택합니다.

| # | KPI | 정의 | 측정 방법 | 목표 |
|---|---|---|---|---|
| 13 | 항목 누락 감소율 | 시나리오 입력의 모든 항목이 응답·구현에 포함된 비율. A군 대비 B군의 응답 항목 누락 비율 감소도. | 시나리오 입력의 항목 목록을 사전 정의 → 응답·git diff와 1:1 매칭 → 충족(✅) / 부분(⚠️) / 미충족(❌) 카운트 | A군 대비 누락 ≥30% 감소, B군 충족률 ≥95% |
| 15 | Constraints 위반율 | L3-B 시나리오 Constraints 카드의 모든 제약(외부 crate 추가, 시그니처 보존, scope 한정) 준수율. | git diff에서 Cargo.toml dependencies / 함수 시그니처 / 파일 scope 변경 정적 검출 + grep 자동화 | B군 위반율 < 5% (시나리오2 실측 0%) |
| 16 | 빌드 통과율 (G1) | 적용된 변경이 빌드 도구(`cargo check`, `npm build` 등)를 통과하는 비율. | CI 자동화 (exit code + warning 카운트) | 100% (하한 기준) |

KPI #14(응답-구현 정합성)는 [[6.Evaluation#6.9.2 후속 평가|§6.9.2]] · [[6.Evaluation#6.9.3 PRD 연계|§6.9.3]] 결정에 따라 프로젝트 범위 외로 취소되었습니다. KPI #14 번호는 번호화 정책에 따라 결번 그대로 유지합니다(취소 이력은 [[DECISIONS]] 참조).

**KPI #13 (항목 누락 감소율) 근거.** 시나리오2 2차 평가에서 H7 가설("L3-B 항목 누락 감소")이 가장 강하게 입증되었습니다. 1차 응답 텍스트 도메인에서 A군 평균 92%, B군 평균 99%이며 B군 분산도 더 작았습니다. 2차 코드 적용 도메인에서도 L3-B가 핵심 3목표를 R1·R2·R3 모두 3/3 충족했습니다. 이는 PromptCraft의 핵심 가치 제안이므로 정식 KPI로 등재합니다.

**KPI #15 (Constraints 위반율) 근거.** 시나리오2 2차 평가에서 H3 가설("L3-B Constraints가 외부 crate 추가를 차단")이 완전 입증되었습니다. A군 11.1% (2/18) 위반, B군 0% (0/18) 위반이며 R1·R2·R3 모두 B군 0%가 일관 관찰되었습니다. PromptCraft의 가장 결정적인 효과 지표이므로 정식 KPI로 등재하며, 후속 평가 사이클의 비결정성을 고려해 목표는 < 5%로 완화합니다.

**KPI #16 (빌드 통과율 G1) 근거.** 시나리오2 2차 평가에서 18/18 PASS (100%)가 실측되었습니다. A·B군 모두 100%로 본 라운드에서 PromptCraft의 차별 효과는 관찰되지 않았으나, 향후 라운드에서 A군 빌드 실패 가능성(특히 광역 scope creep 케이스)을 고려하여 하한 기준 KPI로 등재합니다.

### 7.1.3 KPI 종합표

| 구분 | 지표 수 | 측정 상태 |
|---|---|---|
| 벤치마크 (#1~#2) | 2개 | #1 부분 확정, #2는 종단 연구 별도 설계 필요 |

| **총계** | **15개** | — |

## 7.2 향후 과제 (단계별)

시나리오2 평가에서 도출된 과제와 디자인-구현 격차에서 결정 필요 항목을 통합합니다. P0/P1/P2 3단계 우선순위로 분류합니다.

### 7.2.1 P0 — 평가 운영 (다음 사이클 전제)

현재 P0 과제 없음. 이전 P0-1(모델 식별 메타데이터) 취소 이력은 [[DECISIONS]] 참조. P0 분류 자체는 향후 도입 여지 보존을 위해 유지합니다.

### 7.2.2 P1 — 핵심 기능 개선

PromptCraft 자체 기능 확장과 디자인 결정 항목입니다.

#### A. 평가 자동화 / 기능 확장

| # | 과제 | 설명 | 비고 |
|---|---|---|---|
| P1-3 | **환경 제약 컨텍스트 보강** | `tsconfig`에서 코드 생성 행동에 영향을 주는 옵션(`strict`·`noUncheckedIndexedAccess`·`module`·`target` 등)만 화이트리스트 추출 → 자연어 행동 지침으로 정규화해 `stack-environment` 카드에 투영. eslint(flat config 정적 파싱 불가·보안 원칙 충돌)·`constraints`(사용자 의도 카드, `scanSuggested:false`) 제외. `extends` 상속 머지는 번들 tsconfig API에 위임. | stack-environment 확장. ADR [[DECISIONS]] 참조 |
| P1-4 | **근거 문서화(Rationale) 자동 삽입** | "이 요청을 하는 이유"를 구조화 섹션으로 자동 추가. 모델이 의도를 더 정확히 파악. | — |
| P1-16 | **DB 스키마 마이그레이션 구현** | `PRAGMA user_version` + `src/core/db/migrations/v1.ts` + 단일 트랜잭션 + 실패 시 `~/.promptcraft/db.sqlite.bak.<ts>` 백업 후 기동 중단. 현재 `connection.ts`는 `CREATE TABLE IF NOT EXISTS`만 수행(미구현). [[3.Features\|§3.2.9]] 목표 설계 참조. | SWA S3 데이터 손실 위험 HIGH. ADR [[2026-05-24-feat29-migration-unimplemented]] |

P1-1(응답-구현 정합성 자동 검증)은 [[6.Evaluation#6.9.2 후속 평가|§6.9.2]] · [[6.Evaluation#6.9.3 PRD 연계|§6.9.3]] 결정에 따라 KPI #14와 함께 취소되었습니다. P1-2(검증/가드레일 단계 자동 삽입)는 유일 근거였던 "응답-구현 불일치"가 §6.9.2·§6.9.3에서 PromptCraft 범위 외로 취소되고 [[6.Evaluation|§6 결론]]에서 프롬프트로 해소 불가한 모델 신뢰성 격차로 규정됨에 따라 근거가 소멸하여 취소되었습니다. P1-N 번호는 번호화 정책에 따라 결번 그대로 유지합니다(취소 이력은 [[DECISIONS]] 참조).

#### B. 디자인 시스템 정합성

[[4.UI-Design|§4 UI-Design]]에서 식별된 디자인-구현 격차 중 결정 필요 항목입니다. 현재 해당 분류의 미해결 과제는 없습니다. P1-6(누락 토큰 보강)은 P1-12 Settings UI 구현 실측 결과 신규 토큰이 불필요한 것으로 결론되어 본 표에서 제거되었습니다. P1-N 번호는 번호화 정책에 따라 결번 그대로 유지합니다(실측 결론은 [[2026-05-25-p1-12-settings-ui-done]] 참조).

#### C. UI 변형 / 단축키 확장

UI 변형·단축키 확장 항목입니다. 현재 해당 분류의 미해결 과제는 없습니다. P1-15(empty 카드 editor 측 시각 단서)는 preview-editor 2열 동시 표시 구조에서 우측 PromptPreview가 빈 active 카드를 "(비어 있음 — 출력에서 제외)"로 상시 표시하여 동일 단서를 이미 제공하므로 중복으로 판단되어 제거되었습니다. P1-N 번호는 번호화 정책에 따라 결번 그대로 유지합니다(취소 이력은 [[2026-05-25-p1-15-empty-cue-cancel]] 참조).

P1-11(History UI)은 커밋 `4d1ea9b`로 구현 완료되어 본 표에서 제거되었습니다. P1-N 번호는 번호화 정책에 따라 결번 그대로 유지합니다(완료 이력은 [[2026-05-24-p1-11-history-done]] 참조).

P1-12(Settings UI)는 구현 완료되어 본 표에서 제거되었습니다. "디자인 변형 선택"은 코드·PRD에 정의가 없는 미정의 개념으로 확인되어 범위에서 제외되었으며, "시스템 테마 연동"은 기존 next-themes 구현을 Settings에 명시 노출하는 형태로 흡수되었습니다. P1-N 번호는 번호화 정책에 따라 결번 그대로 유지합니다(완료·결정 이력은 [[2026-05-25-p1-12-settings-ui-done]] 참조).

### 7.2.3 P2 — 후속 기능 확장

평가 사이클과 무관한 독립 기능 확장 항목입니다. 후속 평가 자체는 [[6.Evaluation#6.9.2 후속 평가|§6.9.2]] 결정에 따라 별도 계획하지 않습니다. P2-N 번호는 historical 참조 보존을 위해 결번 그대로 유지합니다(P2-1·P2-2·P2-3·P2-4·P2-5·P2-6·P2-7·P2-9 8건 삭제 이력은 [[DECISIONS]] 참조).

| # | 과제 | 설명 | 의존 |
|---|---|---|---|
| P2-10 | **AI 다듬기 (로컬 LLM)** | Ollama 등 연동 후처리. 비용 제로 원칙 유지하며 품질 향상. | — |

P2-8(i18n 지원 ko/en)은 구현 완료되어 본 표에서 제거되었습니다. 실제 범위는 PRD 정의("카드 레이블·hint·examples")보다 확장되어 카드 정의 전 필드·도메인 오버레이·role-mappings·trees·웹 chrome·CLI·출력 프롬프트 헤더 언어 연동을 포함합니다. P2-N 번호는 번호화 정책에 따라 결번 그대로 유지합니다(완료·범위·known limitations 이력은 [[2026-05-25-p2-8-i18n-done]] 참조).
