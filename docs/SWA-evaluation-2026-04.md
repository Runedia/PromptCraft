## 1. 프로젝트 개요

PromptCraft는 AI 바이브 코딩을 위한 **로컬 웹 기반 구조화 프롬프트 생성 도구**다. 로컬 코드베이스를 스캔·분석하고, SectionCard 모델 기반 웹 UI를 통해 사용자의 의도를 구조화하여 AI 코딩 도구에 최적화된 완성형 프롬프트를 생성한다.

**핵심 설계 철학**: 프롬프트 생성 비용 제로 + 단 한 번의 AI 호출로 완결

```
기존 도구 구조:
  사용자 요청
    → [LLM API 호출 1] 프롬프트 생성/보정  ← 비용 발생
    → [LLM API 호출 2] 실제 작업 수행      ← 비용 발생

PromptCraft 구조:
  사용자 요청
    → [로컬 처리] 스캔 + SectionCard 동적 조립  ← 비용 제로
    → 구조화된 완성 프롬프트를 AI에 투입
    → [LLM API 호출 1회] 실제 작업 수행          ← 비용 발생 지점 단 1회
```

## 2. 장점 분석

| # | 장점 | 근거 |
|---|---|---|
| 1 | **LLM API 제로 비용 구조** | 프롬프트 생성 단계에서 LLM을 일절 사용하지 않음. 경쟁 도구 대비 1회 API 호출 완전 제거 → 이중 비용 구조 해소 |
| 2 | **구조화된 SectionCard 모델** | 25개 카드 기반 동적 조립. 카드 활성/비활성/순서 변경으로 사용자가 프롬프트 구조를 완전 제어. 경쟁 도구의 고정 템플릿보다 유연 |
| 3 | **로컬 코드베이스 스캔 통합** | 언어/프레임워크/구조 자동 감지 + 역할 제안 칩 + 스택 자동 채움. Repomix 같은 별도 도구 없이 단일 도구에서 해결 |
| 4 | **크로스모델 호환** | Claude, GPT, Gemini 등 주요 AI 코딩 도구의 프롬프팅 원칙을 카드 hint/examples에 내재화. 특정 모델에 종속되지 않음 |
| 5 | **깔끔한 계층 분리** | core/server/web/shared/cli 5개 레이어로 관심사 분리. TypeScript strict 모드, 순수 불변 카드 조작 함수 |
| 6 | **실시간 Dual-Pane UX** | CLI 위자드의 순차성 제약을 해결. 카드 편집과 마크다운 프리뷰 동시 확인, 드래그앤드롭, Undo/Redo 10단계 |
| 7 | **보안 기반 설계** | CSP, CORS localhost 제한, 경로 순회 차단, .env/바이너리 차단. 로컬 도구임에도 보안 미들웨어 3종 구비 |
| 8 | **@file 멘션 시스템** | 서버 사이드 경로 자동완성 + 파일 내용 인라인 삽입. 컨텍스트 주입의 진입 장벽을 낮춤 |

## 3. 단점 분석

| # | 단점 | 심각도 | 근거 |
|---|---|---|---|
| 1 | **시장 타이밍 리스크** | 높음 | 2026년 AI 코딩 도구(Claude Code, Cursor, Copilot)가 자체 컨텍스트 관리를 강화하는 추세. 프롬프트 구조화의 필요성 자체가 줄어들 수 있음 |
| 2 | **테스트 커버리지 부족** | 높음 | API 라우트 테스트 0개, UI 컴포넌트 테스트 0개, E2E 테스트 부재. 10개 유닛 테스트만 존재 |
| 3 | **사용자 검증 부재** | 높음 | PRD의 "One-Shot 성공률 극대화" 주장이 정량적 데이터로 뒷받침되지 않음. 실제 토큰 절감량/성공률 벤치마크 없음 |
| 4 | **워크플로우 마찰** | 중간 | `promptcraft serve` → 브라우저 → 카드 작성 → 복사 → AI 도구에 붙여넣기. 단계가 많아 "바이브 코딩"의 즉시성과 충돌 |
| 5 | **25개 카드의 인지 부하** | 중간 | 카드 수가 많아 초보 사용자에게 압도적. 트리별 기본 활성 4~5개로 완화했으나 카드 풀의 선택지가 과다 |
| 6 | **DB 스키마 관리 부재** | 중간 | 마이그레이션, 버전 관리, 트랜잭션 처리 미구현. SQLite 단일 사용자 환경이라 당장은 문제없으나 확장성 제약 |
| 7 | **한국어 중심 설계** | 낮음 | 카드 레이블, hint, examples가 한국어 기준. i18n 미구현으로 글로벌 확장에 추가 비용 필요 |
| 8 | **AI 다듬기 기능 P2 방치** | 낮음 | "선택적 AI 다듬기"가 로드맵에만 존재. 경쟁 도구들이 이미 AI 기반 프롬프트 개선을 제공 |

## 7. 이해관계자 피드백 #1 — "불완전한 프롬프트 문제의 현재 유효성"

> **질의**: 불완전한 질문으로 인해 AI의 잘못된 이해와 작업으로 인한 토큰 비효율 및 시간 소모 등, 이러한 관점에서 봤을 때에는 어떠한가?

### 7.1 문제의 현재 유효성

| 관점 | 2024년 | 2026년 4월 | 변화 방향 |
|---|---|---|---|
| 불완전한 프롬프트로 인한 재질문 | 매우 빈번 | 여전히 발생 | 모델 능력 향상으로 빈도 감소, 그러나 복잡한 태스크에서는 여전히 핵심 병목 |
| 토큰 비용 민감도 | 높음 | **더 높아짐** | Opus급 모델의 토큰 단가 상승 + 100K+ 컨텍스트 사용 일반화로 누적 비용 증가 |
| 컨텍스트 윈도우 오염 | 심각 | 여전히 심각 | 재질문 왕복마다 이전 대화 전체가 재전송되는 구조는 불변 |
| AI 도구의 자체 완화 | 미약 | **상당히 개선** | Claude Code의 CLAUDE.md, Cursor의 .cursorrules, Copilot의 instructions.md 등 |

**정량적 논거** — 단일 재질문 왕복의 실제 비용 추정:

```
1회 왕복 비용 = (이전 대화 컨텍스트 + 새 질문) × 입력 토큰 단가
             + AI 응답 × 출력 토큰 단가

예시 (Opus급, 3회 왕복):
  1회차: 2K 입력 + 4K 출력 = 6K 토큰
  2회차: 6K 누적 + 2K 질문 + 4K 출력 = 12K 토큰
  3회차: 12K 누적 + 2K 질문 + 4K 출력 = 18K 토큰
  → 총 36K 토큰 (이상적 One-Shot 6K 대비 6배)
```

이 구조적 비효율은 모델 세대와 무관하게 존재한다. **문제 자체는 확실히 유효하다.**

### 7.2 해법의 경쟁 구도 변화

| 해법 | 마찰 | 커버리지 | 비용 |
|---|---|---|---|
| **PromptCraft** (외부 도구) | 높음 — 브라우저 전환 + 카드 작성 + 복사 붙여넣기 | 높음 — 25개 섹션 세밀 제어 | 제로 |
| **CLAUDE.md / .cursorrules** (규칙 파일) | 낮음 — 한 번 작성하면 자동 적용 | 중간 — 프로젝트 수준 고정 컨텍스트 | 제로 |
| **Claude Code의 /compact + 멘션** | 제로 — 이미 도구 안에 있음 | 중간 — @파일 멘션, 자동 컨텍스트 | 제로 |
| **Cursor Composer** | 제로 — 에디터 통합 | 중간 — 코드베이스 인덱싱 | 구독료 |

### 7.3 PromptCraft가 여전히 우위를 가지는 정확한 니치

| 시나리오 | PromptCraft 우위 | 이유 |
|---|---|---|
| 대규모 기능 구현 (100+ 라인) | **강함** | 역할/스택/범위/제약/수락기준을 구조화하면 One-Shot 확률이 실질적으로 올라감 |
| 복잡한 에러 디버깅 | **강함** | 에러 증거 + 시도한 방법 + 빌드 로그를 체계적으로 제공하면 컨텍스트 오염 대폭 감소 |
| 간단한 질문/수정 | **약함** | 카드 작성 오버헤드가 직접 질문보다 큼 |
| 이미 CLAUDE.md가 있는 프로젝트 | **약함** | 프로젝트 컨텍스트가 이미 자동 주입됨 |
| API 비용에 극도로 민감한 사용자 | **강함** | 재질문 1회 차단의 금전적 가치가 명확 |

### 7.4 초기 종합 판단

**조건부 유효.** 핵심 가치 제안은 논리적으로 타당하나, 두 가지 구조적 리스크 존재:

1. **시장 수렴 리스크**: AI 코딩 도구가 자체 컨텍스트 관리와 프롬프트 최적화를 내재화하는 추세. "프롬프트를 잘 써야 한다"는 전제 자체가 약화되는 방향.
2. **워크플로우 마찰**: "브라우저에서 프롬프트 작성 → 클립보드 복사 → AI 도구에 붙여넣기"라는 3단계가, AI 도구의 인라인 컨텍스트 주입(CLAUDE.md, .cursorrules 등)보다 마찰이 큼.

초기 생존 전략으로 **독립 도구 → MCP 서버**로의 전환을 제안했다.

## 8. 이해관계자 피드백 #2 — "CLAUDE.md의 한계와 MCP 비용 문제"

> **피드백 1**: CLAUDE.md가 작성되어 있어도 매번 의식하고 구조적인 프롬프트를 입력하기 어렵기 때문에 이 프로젝트를 계획한 것이다.
>
> **피드백 2**: MCP의 경우 결국 LLM API의 토큰을 사용하도록 되어 있다.
>
> **피드백 3**: VS Code 또는 Antigravity의 확장 도구는 적극 고려해보겠다.

### 8.1 CLAUDE.md의 한계 — "존재"와 "활용"은 다르다

이 지적은 정확하다. CLAUDE.md는 **AI 도구 측의 컨텍스트**를 자동화하지만, **사용자 측의 질문 품질**은 자동화하지 않는다.

```
CLAUDE.md가 해결하는 것:
  "이 프로젝트는 Next.js 14, pnpm, TypeScript strict를 사용한다"
  → AI가 이미 알고 있음 (자동 주입)

CLAUDE.md가 해결하지 못하는 것:
  "에러 나는데 고쳐줘"  ← 사용자의 입력 습관
  vs
  "Header.tsx:42에서 hydration mismatch 발생.
   시도: useEffect 래핑 → 실패. 기대 동작: SSR과 클라이언트 렌더 일치"
```

PromptCraft의 SectionCard는 이 간극을 **강제적 구조화**로 메운다. "에러 증거" 카드가 비어 있으면 프리뷰에서 회색으로 표시되고, 사용자는 채워야 한다는 시각적 압박을 받는다. 이것은 CLAUDE.md가 제공할 수 없는 **질문 행동의 교정 장치**다.

| 도구 | AI 컨텍스트 자동화 | 사용자 질문 구조화 | 행동 교정 |
|---|---|---|---|
| CLAUDE.md | 해결 | 미해결 | 없음 |
| PromptCraft | 해결 (스캔) | **해결 (카드 강제)** | **있음 (빈 카드 시각화)** |
| 직접 타이핑 | 수동 | 사용자 역량 의존 | 없음 |

**이 관점에서 PromptCraft의 가치 제안은 "토큰 절감 도구"가 아니라 "프롬프트 습관 교정 도구"에 가깝다.** 이 재정의가 더 정직하고 방어 가능한 포지셔닝이다.

### 8.2 MCP 서버 = LLM 토큰 소비 — 비용 제로 원칙 위반

MCP 서버로 전환하면 PromptCraft의 핵심 원칙("프롬프트 생성에 LLM 비용 제로")이 무너진다.

```
MCP 서버 모드:
  사용자 → Claude Code → MCP 호출 → PromptCraft 엔진
  → 결과가 Claude Code 컨텍스트에 주입 ← 토큰 소비 발생
  → Claude가 구조화된 프롬프트를 "읽는 데" 토큰을 소비

독립 도구 모드 (현재):
  사용자 → PromptCraft 웹 UI → 프롬프트 생성 ← 토큰 제로
  → 완성된 프롬프트를 AI에 투입 ← 1회만 소비
```

**MCP 서버 제안을 철회한다.** 독립 도구로서의 비용 제로 원칙이 차별점이라면, 그것을 지키는 것이 올바른 전략이다.

### 8.3 수정된 개선 제안

| # | 이전 제안 | 수정 제안 | 이유 |
|---|---|---|---|
| 1 | MCP 서버 모드 | **VS Code / Antigravity 확장** — 사이드바 패널에서 카드 작성 → 클립보드 복사 | LLM 토큰 미사용 원칙 유지 + 브라우저 전환 마찰 제거 |
| 2 | "고비용 태스크 전용" 포지셔닝 | **"프롬프트 습관 교정 도구"** 포지셔닝 | 기술적 가치(토큰 절감)보다 행동적 가치(구조화 습관 형성)가 더 방어 가능 |
| 3 | A/B 벤치마크만 제안 | A/B 벤치마크 유지 + **"학습 곡선 측정"** 추가 | 사용 횟수 증가에 따라 카드 없이도 구조적 질문을 하게 되는지 측정하면, "교정 도구"로서의 효과 입증 가능 |

## 9. 이해관계자 피드백 #3 — "Run as... 직접 실행 기능"

> **제안**: 어차피 로컬에서 실행되고 target 경로도 알고 있으니, 생성한 프롬프트로 해당 AI를 호출해 바로 실행하도록 기능을 추가하는 것은 어떤가? 예를 들어 `Run as Claude Code`, `Run as Copilot` 등의 버튼을 만들어서 누르면 해당 프로젝트로 CWD를 한 후 작성된 프롬프트를 실행하는 것.

### 9.1 워크플로우 마찰의 완전 해소

```
현재:   PromptCraft → [복사] → [에디터 전환] → [붙여넣기] → [실행]
제안:   PromptCraft → [Run as Claude Code] → 끝
```

**비용 제로 원칙이 깨지지 않는다.** PromptCraft는 프롬프트 생성까지만 담당하고, LLM 호출은 외부 도구가 수행한다. 역할 분리가 명확하다.

### 9.2 기술적 실현 가능성

| AI 도구 | 호출 방식 | 실현 가능성 | 비고 |
|---|---|---|---|
| **Claude Code** | `claude -p "프롬프트" --cwd /target/path` | 높음 | CLI가 파이프 입력과 `--cwd` 플래그를 지원 |
| **Copilot CLI** | `gh copilot suggest "프롬프트"` | 중간 | GitHub CLI 확장, 구조화된 장문 프롬프트 전달에 제약 가능 |
| **Cursor** | 직접 CLI 호출 불가 | 낮음 | API/CLI 미공개. `.cursorrules`에 쓰고 Cursor를 여는 우회 가능 |
| **Aider** | `aider --message "프롬프트"` | 높음 | CLI 네이티브, `--cwd` 지원 |
| **터미널 직접** | 표준 출력으로 파이프 | 높음 | 범용 폴백 |

### 9.3 구현 구조

```
[Run as...] 버튼 클릭
  → POST /api/prompt/run
    body: { provider: "claude-code", prompt: "...", cwd: "/target/path" }
  → 서버: child_process.spawn("claude", ["-p", prompt, "--cwd", cwd])
  → 결과 스트리밍 or 프로세스 분리(detach) 후 알림
```

서버가 이미 `cwd`(스캔 경로)를 알고 있고, 프롬프트 빌드 결과도 서버 측에서 조립된다. **추가 인프라가 거의 필요 없다.**

### 9.4 주의사항

| 리스크 | 심각도 | 완화 |
|---|---|---|
| 장시간 실행 프로세스 관리 | 중간 | detach 모드로 분리 후 PID 반환. PromptCraft는 "발사 후 망각" |
| 프롬프트 길이 제한 (셸 인자) | 중간 | 임시 파일에 프롬프트 저장 → `claude -p "$(cat /tmp/prompt.md)"` 또는 `--file` 플래그 활용 |
| 보안 (임의 명령 실행) | 높음 | provider 화이트리스트 고정. 사용자 정의 명령어는 config에서만 등록 가능하게 제한 |
| AI 도구 미설치 | 낮음 | `which claude` 등으로 사전 확인, 미설치 시 버튼 비활성화 + 안내 |

### 9.5 포지셔닝 전환 효과

```
이전:  프롬프트 "생성" 도구  (출력물: 텍스트)
이후:  프롬프트 "설계 → 실행" 도구  (출력물: AI 작업 결과)
```

**"프롬프트를 잘 만드는 도구"에서 "AI 코딩을 잘 시작하는 도구"로 전환된다.** 복사-붙여넣기 마찰이 제거되면, 카드 작성 시간만이 유일한 비용이 되고, 그 비용은 One-Shot 성공률 향상으로 정당화된다.

에디터 확장보다 **구현 비용이 낮고 효과는 동등하다.** 우선순위를 에디터 확장보다 높게 설정할 수 있다.

## 10. 최종 종합 판단

### PromptCraft의 핵심 가치 — 3겹 구조

1. **비용 구조**: 프롬프트 생성에 LLM 제로 — MCP 시대에도 유효한 차별점
2. **행동 교정**: 카드 시스템이 사용자에게 구조적 사고를 강제 — CLAUDE.md가 대체 불가
3. **컨텍스트 집약**: 스캔 + 멘션으로 분산된 정보를 한 곳에 조립

### 수정된 전략 로드맵

| 우선순위 | 항목 | 효과 |
|---|---|---|
| **P0** | `Run as...` 직접 실행 기능 | 워크플로우 마찰 완전 제거. 구현 비용 낮음 (서버 이미 cwd + 프롬프트 보유) |
| **P0** | A/B 벤치마크 + 학습 곡선 측정 | 가치 제안 정량적 입증. "습관 교정 도구"로서의 효과 데이터 |
| **P1** | API 라우트 + E2E 테스트 | MVP 안정성 확보. supertest 인프라 이미 존재 |
| **P1** | VS Code / Antigravity 확장 | 에디터 통합으로 브라우저 전환 제거 (Run as... 이후 단계) |
| **P2** | 포지셔닝 재정의: "프롬프트 습관 교정 도구" | 마케팅 메시지 전환 |
| **P2** | i18n, 스캔 캐싱, DB 마이그레이션 | 글로벌 확장 + 기술 부채 해소 |

### 최종 결론

불완전한 프롬프트로 인한 토큰 비효율 문제는 **2026년에도 구조적으로 유효**하다. CLAUDE.md 등 AI 도구의 자체 컨텍스트 관리가 개선되었으나, 이는 "AI가 프로젝트를 이해하는 것"만 해결할 뿐 "사용자가 구조적으로 질문하는 것"은 해결하지 못한다.

PromptCraft의 위협은 "도구의 존재 가치"가 아니라 **"워크플로우 마찰"**으로 좁혀진다. `Run as...` 직접 실행 기능과 에디터 확장이 이 마찰을 해소하면, **"비용 제로 프롬프트 설계 → 원클릭 AI 실행"**이라는 고유한 포지셔닝이 확보된다.
