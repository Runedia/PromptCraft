# PromptCraft 2.3 발표 대본

**총 발표 시간**: 20분  
**슬라이드 수**: 16장  
**슬라이드당 평균**: 약 75초

---

## Slide 01 — 타이틀 (30초)

안녕하세요. 오늘은 PromptCraft 프로젝트 계획을 발표하겠습니다.

PromptCraft는 AI 바이브 코딩을 위한 로컬 웹 기반 구조화 프롬프트 생성 도구입니다.

오늘 발표에서는 이 도구가 왜 필요한지, 어떻게 동작하는지, 그리고 어떤 기술로 어떻게 만들지를 순서대로 말씀드리겠습니다.

---

## Slide 02 — 문제 정의 (2분)

먼저 왜 이 도구가 필요한지부터 시작하겠습니다.

AI 코딩 도구를 쓰다 보면 공통적으로 세 가지 문제가 반복됩니다.

첫 번째는 **잦은 재질문**입니다. AI가 개발자의 의도를 파악하지 못해 "혹시 이런 뜻인가요?", "좀 더 구체적으로 말씀해 주시겠어요?" 같은 핑퐁이 계속됩니다.

두 번째는 **토큰 낭비**입니다. 재질문이 한 번 오갈 때마다 대화 컨텍스트 전체가 다시 전송됩니다. 비용이 누적됩니다.

세 번째는 **엉뚱한 결과물**입니다. "렌더링 에러 고쳐줘" 한 줄로는 AI가 어떤 환경인지, 에러가 어떻게 생겼는지 알 수 없습니다.

그런데 이 문제를 해결하겠다는 기존 프롬프트 보조 도구들도 사실은 이중 비용 구조를 가집니다. 사용자 요청을 받으면 LLM API를 한 번 호출해서 프롬프트를 생성하고, 그 프롬프트로 다시 LLM을 호출합니다. API 비용이 두 배입니다.

PromptCraft는 이 구조를 근본적으로 다르게 접근합니다.

---

## Slide 03 — 해결 전략 (1분 30초)

PromptCraft의 핵심은 단순합니다. **프롬프트 생성 단계에서 LLM을 쓰지 않는 것**입니다.

왼쪽이 기존 도구입니다. 사용자 요청이 들어오면 LLM을 두 번 호출합니다. 비용이 두 배, 실패 시에는 그 이상입니다.

오른쪽이 PromptCraft입니다. 사용자 요청이 들어오면 로컬에서 프로젝트를 스캔하고, SectionCard 모델로 프롬프트를 조립합니다. 이 과정에서 LLM은 전혀 사용하지 않습니다. 그리고 완성된 프롬프트를 AI에 단 한 번만 투입합니다.

이게 가능한 이유가 아래 인과 구조입니다.

역할, 목표, 컨텍스트, 제약이 섹션으로 명확히 분리된 프롬프트를 받으면, AI 모델은 "이게 무슨 뜻이지?"를 고민하는 단계를 생략할 수 있습니다. 사고 토큰이 전량 문제 해결에 집중됩니다. 결과적으로 한 번에 정확한 답을 받을 확률이 높아지고, 재질문 왕복이 사라집니다.

---

## Slide 04 — PRD 2.2 → 2.3 전환 (1분 30초)

이제 이전 버전인 PRD 2.2와 현재 2.3이 어떻게 다른지 보겠습니다.

PRD 2.2는 CLI Ink 위자드 방식이었습니다. 터미널에서 `promptcraft build`를 실행하면 순서대로 질문이 나오고, 답변하면 다음 질문이 오는 구조입니다.

이 방식에는 구조적 제약이 있었습니다. 네 가지입니다.

이전 단계로 돌아갈 수 없습니다. 어떤 섹션이 필요한지 미리 알 수 없습니다. 결과를 작성하면서 동시에 볼 수 없습니다. 그리고 터미널은 레이아웃 자유도가 낮습니다.

PRD 2.3에서는 이 모든 제약을 해결했습니다. 플랫폼을 로컬 웹 서버로 전환했고, Handlebars 고정 템플릿 대신 SectionCard 동적 조립 모델을 도입했습니다. 진입 명령어는 `promptcraft serve`입니다.

---

## Slide 05 — 시스템 아키텍처 (1분 30초)

시스템 구조는 3계층으로 되어 있습니다.

**1계층은 브라우저**입니다. React 19와 Vite 6로 구성된 웹 앱입니다. TreeSelect, WorkspacePage, SectionCard, PromptPreview 같은 컴포넌트들이 여기 있습니다. 상태 관리는 Zustand와 Zundo를 씁니다.

**2계층은 Express API 서버**입니다. localhost:3000에서 실행되며 9개 REST API 라우트를 제공합니다. browse, scan, trees, cards, prompt, history, templates, config, mention입니다. CSP 헤더, localhost CORS, 경로 순회 방지 미들웨어가 보안을 담당합니다.

**3계층은 코어 엔진**입니다. Node.js로 작성된 순수 비즈니스 로직입니다. 스캐너, CardSession, promptBuilder, tokenEstimator, SQLite DB가 여기 있습니다.

중요한 점은 **모든 데이터가 localhost 안에서만 처리**된다는 것입니다. 외부 네트워크 접근은 완전히 차단됩니다.

## Slide 06 — SectionCard 모델 (1분 30초)

PromptCraft의 핵심 데이터 모델인 SectionCard를 설명하겠습니다.

프롬프트의 각 섹션이 독립적인 카드 단위로 존재합니다. 왼쪽이 활성 카드입니다. 보라색 테두리로 "현재 프롬프트에 포함됨"을 나타냅니다. 오른쪽이 비활성 카드입니다. 점선 테두리로 카드 풀에 대기 중임을 나타냅니다.

카드가 가진 속성은 id, label, required, active, order, inputType, value, template, defaultValue 등입니다.

프롬프트 빌드 규칙은 간단합니다. `active === true`이고 `value.trim() !== ''`인 카드만, `order` 오름차순으로 조립합니다. 빈 카드는 미리보기에서 회색으로 표시되지만, 실제 출력에는 포함되지 않습니다.

전체 카드 정의는 25개이며, 입력 타입은 text, multiline, select, select-or-text, multiline-mention 5가지입니다.

---

## Slide 07 — 4개 워크플로우 트리 (1분 30초)

사용자는 상황 유형에 따라 4개 워크플로우 트리 중 하나를 선택합니다.

**에러 해결**은 런타임 에러, 빌드 에러 등 디버깅 상황입니다. 기본적으로 role, goal, stack-environment, error-evidence, tried-methods 5개 카드가 활성화됩니다.

**기능 구현**은 신규 기능 추가나 기존 기능 수정입니다. role, goal, stack-environment, impl-scope가 기본 활성입니다.

**코드 리뷰**는 보안, 성능, 가독성 점검입니다. role, goal, review-code, review-focus가 기본입니다. goal 카드에는 기본값이 자동으로 채워집니다.

**개념 학습**은 기술 개념 이해와 적용입니다. role, goal, concept, skill-level, output-pref가 기본입니다.

각 트리는 카드 오버라이드 메커니즘으로 카드별 기본값, 힌트, 예시를 상황에 맞게 재정의합니다. 예를 들어 코드 리뷰의 output-format 카드는 코드 리뷰 전용 4가지 옵션을 제공합니다.

## Slide 08 — Dual-Pane UI (1분 30초)

UI 레이아웃을 보겠습니다.

화면이 60:40으로 나뉩니다. 왼쪽은 카드 편집기, 오른쪽은 실시간 마크다운 프리뷰입니다.

왼쪽에서 카드를 선택하고 값을 입력하면, 오른쪽 프리뷰가 즉시 갱신됩니다. 카드 순서는 드래그 앤 드롭으로 변경할 수 있고, @dnd-kit 라이브러리를 사용합니다.

불필요한 카드는 X 버튼으로 카드 풀로 내보낼 수 있습니다. 반대로 카드 풀에서 필요한 카드를 추가할 수 있습니다.

Undo/Redo는 Zundo temporal 미들웨어로 10단계까지 지원합니다. 키보드 단축키도 지원합니다. Ctrl+Enter로 클립보드에 복사, Ctrl+S로 템플릿 저장, Ctrl+Z로 실행 취소입니다.

프리뷰 하단에는 토큰 추정값이 실시간으로 표시됩니다. 한국어와 영어를 구분해서 추정합니다.

## Slide 09 — Pre-scan & 역할 자동 제안 (1분)

스캔 통합 기능을 보겠습니다.

프로젝트 경로를 입력창에 타이핑하면, 별도로 스캔 버튼을 누를 필요가 없습니다. 800ms 디바운스 후 자동으로 POST /api/scan이 실행됩니다.

스캔이 완료되면 두 가지가 자동으로 처리됩니다.

첫째, stack-environment 카드에 감지된 기술 스택이 자동으로 채워집니다. "Next.js 14, TypeScript, pnpm" 같은 정보입니다.

둘째, 감지된 프레임워크를 기반으로 역할 제안 칩이 최대 5개 표시됩니다. React가 감지되면 "React 개발자", "프론트엔드 개발자" 같은 칩이 나타납니다. 클릭하면 바로 role 카드에 입력됩니다.

## Slide 10 — @file 멘션 & 보안 (1분 30초)

@file 멘션 시스템과 보안을 함께 설명하겠습니다.

`multiline-mention` 타입의 카드에서 `@`를 입력하면 서버 사이드 파일 경로 자동완성이 작동합니다. `@src/components/`까지 입력하면 해당 디렉토리의 파일 목록이 드롭다운으로 나타납니다. 키보드 화살표로 탐색하고 Enter로 선택하면, 파일 내용이 카드에 인라인으로 삽입됩니다.

폴더 브라우저 모달도 있어서 GUI로 파일시스템을 탐색할 수 있습니다. Windows 드라이브 문자도 지원합니다.

보안 미들웨어는 세 가지입니다.

CSP 헤더로 XSS를 방지합니다. CORS는 localhost만 허용합니다. pathGuard는 경로 순회 시도, 즉 `../` 공격과 .env 파일 접근을 차단합니다.

## Slide 11 — 기술 스택 (1분 30초)

기술 스택을 세 계층으로 정리했습니다.

**공통 계층**은 CLI와 웹 모두에서 사용합니다. TypeScript 5.9+, Node.js 24+, pnpm 10+, better-sqlite3, Commander.js, Biome입니다.

**웹 전용 계층**이 이번 2.3에서 새로 추가된 부분입니다. Express 5, Vite 6, React 19, Tailwind CSS v4, shadcn/ui가 UI를 담당합니다. 상태 관리는 Zustand 5와 Zundo 2, 드래그 앤 드롭은 @dnd-kit, 프리뷰 렌더링은 react-markdown을 사용합니다.

**레거시 계층**은 P2 우선순위로 유지보수 모드입니다. Ink 6, Handlebars 4, @inquirer/prompts로 구성된 기존 CLI 위자드입니다. 완전히 제거하지 않고 선택적으로 사용할 수 있도록 유지합니다.

## Slide 12 — 비기능 요구사항 (1분)

성능과 보안 기준입니다.

프리뷰 반응속도는 100ms 이내입니다. 카드를 입력하는 즉시 오른쪽 프리뷰가 갱신되어야 합니다.

스캔 완료는 5초 이내입니다.

네트워크는 localhost로 격리됩니다. 외부에서 접근할 수 없습니다.

Undo/Redo는 10단계를 지원합니다.

경로 순회 차단율은 100%입니다. pathGuard 미들웨어가 403으로 응답합니다.

세션 복구는 localStorage로 자동 저장됩니다. 브라우저를 실수로 닫아도 이전 작업을 이어갈 수 있습니다.

반응형 레이아웃은 4단계로 나뉩니다. 1280px 이상이 기본 Dual-Pane입니다.

## Slide 13 — 개발 타임라인 (1분)

4개월 로드맵입니다.

**1개월차**는 웹 인프라 구축입니다. Express 서버와 Vite 설정, SectionCard 타입 정의, 25개 카드 JSON 작성, REST API 스켈레톤을 완성합니다.

**2개월차**는 코어 웹 UI MVP입니다. Dual-Pane 레이아웃, TreeSelect 화면, 카드 편집기, 실시간 프리뷰, Zustand 스토어를 완성합니다. 이 시점에서 기본적인 프롬프트 생성 플로우가 동작해야 합니다.

**3개월차**는 고급 기능 완성입니다. @file 멘션, 폴더 브라우저, 드래그 앤 드롭, Pre-scan, 역할 자동 제안, 보안 미들웨어를 추가합니다.

**4개월차**는 안정화와 릴리스입니다. 반응형 레이아웃, 세션 복구, 히스토리/템플릿 연동, 테스트 보강, 문서화, 최종 릴리스입니다.

## Slide 14 — 향후 과제 (1분)

앞으로 구현 예정인 세 가지 과제입니다.

첫 번째는 **Rationale 자동 삽입**입니다. "이 요청을 하는 이유"를 구조화 섹션으로 자동 추가하는 기능입니다. AI가 단순히 뭘 해야 하는지가 아니라 왜 해야 하는지를 알면, 더 맥락에 맞는 답을 줍니다.

두 번째는 **Defensive Thinking 가드레일**입니다. 모든 프롬프트 빌드가 완료되면, "이 코드의 잠재적 문제점 3가지와 방지 방법을 함께 언급해 주세요" 같은 검증 지시어를 선택적으로 삽입하는 기능입니다. AI에게 완벽주의적 사고를 요구해서 결과물 신뢰도를 높입니다.

세 번째는 **자동 Constraints 매핑**입니다. `tsconfig.json`, `.eslintrc` 같은 설정 파일에서 제약 조건을 자동으로 추출해 constraints 카드에 채우는 기능입니다.

## Slide 15 — 성공 지표 (1분)

프로젝트 성공을 판단할 KPI(핵심 성과 지표)입니다.

**70% 시간 단축** — 수동으로 프롬프트를 작성하는 것보다 70% 이상 빠르게 완성 프롬프트를 만들 수 있어야 합니다.

**빈 헤더 출력 Zero** — 값이 비어있는 active 카드는 절대 프롬프트에 포함되어선 안 됩니다. 단위 테스트로 검증합니다.

**프리뷰 갱신 100ms 이내** — 카드 입력 이벤트부터 DOM 갱신까지 100ms를 넘지 않아야 합니다.

**LLM 호출 Zero** — 코어 플로우, 즉 스캔부터 프롬프트 빌드까지 외부 API 호출이 없어야 합니다.

**TypeScript 컴파일 에러 Zero** — 빌드 단계에서 에러가 없어야 합니다.

**경로 순회 차단 100%** — pathGuard 테스트에서 모든 시도가 403으로 차단되어야 합니다.
