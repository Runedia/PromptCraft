# PRD — PromptCraft 2.1

**AI 바이브 코딩을 위한 로컬 설치형 구조화 프롬프트 생성 도구**

---

## 1. 개요

### 1.1 배경

AI 코딩 도구(Claude, Gemini, ChatGPT 등)를 활용한 '바이브 코딩(Vibe Coding)'의 성공 여부는 사용자가 제공하는 프롬프트의 품질에 직결됩니다. 그러나 대부분의 개발자는 모호하고 불완전한 질문을 입력하여 다음 세 가지 문제를 반복합니다.

- **잦은 재질문**: 의도 파악에 실패한 AI와의 불필요한 핑퐁
- **토큰 낭비**: 재질문 왕복마다 발생하는 컨텍스트 누적 비용
- **엉뚱한 결과물**: 구체성이 부족한 입력으로 인한 저품질 코드 생성

**기존 AI 프롬프트 보조 도구들의 구조적 문제:**

시장에 출시된 대부분의 프롬프트 보조 도구들은 프롬프트 생성 및 보정 자체에 LLM API를 사용합니다. 즉 이중 비용 구조를 가집니다.

```
기존 도구 구조:
  사용자 요청
    → [LLM API 호출 1] 프롬프트 생성/보정  ← 비용 발생
    → 생성된 프롬프트를 AI에 투입
    → [LLM API 호출 2] 실제 작업 수행      ← 비용 발생
    → (One-Shot 실패 시 추가 왕복)           ← 추가 비용 발생
```

PromptCraft는 이 구조를 근본적으로 해결합니다. 프롬프트 생성 단계에서 LLM을 일절 사용하지 않고, 단일 AI 호출에서 One-Shot 성공을 이끌어내는 구조화된 프롬프트를 로컬에서 완성합니다.

```
PromptCraft 구조:
  사용자 요청
    → [로컬 처리] 스캔 + Q&A + 템플릿 조립  ← 비용 제로
    → 구조화된 완성 프롬프트를 AI에 투입
    → [LLM API 호출 1회] 실제 작업 수행      ← 비용 발생 지점 단 1회
```

### 1.2 제품 정의

PromptCraft는 로컬 코드베이스를 스캔·분석하고, 분기형 Q&A를 통해 사용자의 의도를 구조화하여 AI 코딩 도구에 최적화된 완성형 프롬프트를 생성하는 **로컬 설치형 Node.js CLI/웹 기반 도구**입니다.

**핵심 설계 철학: 프롬프트 생성 비용 제로 + 단 한 번의 AI 호출로 완결**

| 구분 | 방식 | 설명 |
|---|---|---|
| 프로젝트 스캔 | 규칙 기반 (로컬) | 파일 시스템 분석, 언어/프레임워크 자동 감지 |
| 분기 Q&A | 규칙 기반 (로컬) | JSON으로 사전 정의된 의도 파악 분기 트리 |
| 프롬프트 조립 | 규칙 기반 (로컬) | 구조화 템플릿에 수집 데이터 자동 결합 |
| 프롬프트 다듬기 | 선택적 AI (옵션) | 필요 시 로컬 LLM을 통한 후처리 |

### 1.3 One-Shot 성공률과 토큰 효율의 인과 관계

PromptCraft의 토큰 최적화는 단순한 입력 압축이 아닙니다. 실제 효과는 다음 인과 구조에서 발생합니다.

```
구조화된 프롬프트 (역할·목표·컨텍스트·제약이 섹션으로 명시 분리)
    ↓
모델이 의도 파악 단계를 생략
    ↓
사고 토큰(Thinking Tokens)이 전량 문제 해결에 집중
    ↓
One-Shot 응답 성공률 상승
    ↓
재질문 왕복 제거
    ↓
총 API 호출 비용 최소화
```

**절감되는 토큰의 실질 비중:**

| 절감 유형 | 상대적 비중 | 설명 |
|---|---|---|
| 입력 압축 (스캔 요약) | 낮음 | 파일 트리 전체 → 핵심 경로 요약 |
| 재질문 왕복 제거 | 높음 | 대화 1회 = 누적 컨텍스트 전체 재전송 |
| 프롬프트 생성 단계 API 제거 | 높음 | 경쟁 도구 대비 1회 API 호출 완전 제거 |

### 1.4 구조화 템플릿의 설계 근거

Anthropic의 Agent Skills(SKILL.md) 포맷 및 Claude Prompting Best Practices에서 도출된 원칙을 프롬프트 템플릿에 내재화합니다. 이 포맷은 Claude Code뿐 아니라 GPT Codex, Gemini 등 주요 AI 코딩 도구에서 크로스모델 표준으로 수렴 중입니다.

**핵심 원칙:**
- 역할·목표·컨텍스트·제약을 명시적으로 섹션 분리
- 불변 정보(스택/환경)를 상단 배치, 가변 정보(현재 문제)를 하단 배치
- 자연어 산문 대신 구조적 헤더로 정보 레이블링
  → 동일 정보를 산문 대비 적은 토큰으로, 더 높은 해석 정확도로 전달

### 1.5 목표

- **프롬프트 생성 단계의 LLM 비용 완전 제거** (로컬 규칙 기반 100% 동작)
- **One-Shot 응답 성공률 극대화** (구조화 템플릿을 통한 모델 사고 효율 향상)
- **재질문 왕복 및 불필요한 토큰 소비 최소화**
- 프로젝트 컨텍스트를 매번 수동으로 설명해야 하는 반복 작업 제거
- 빠르고 가벼운 100% Node.js / JavaScript 생태계 기반 동작 보장

### 1.6 프로젝트 정보

| 항목 | 내용 |
|---|---|
| 팀 구성 | 최대 4인 |
| 개발 환경 | Node.js 24+, pnpm |
| 개발 기간 | 3 ~ 4개월 |
| 주요 언어 | TypeScript / JavaScript (Node.js, React) |

## 2. 사용자 정의

### 2.1 타겟 사용자

**Primary** — AI 코딩 도구를 자주 사용하며 바이브 코딩을 실천하려는 개발자

- AI에게 상황 설명을 매번 타이핑하는 것이 귀찮은 사람
- 질문을 구체화하지 못해 AI와 불필요한 스무고개를 자주 하는 사람
- LLM API 비용 또는 토큰 소모량에 민감한 사람
- 프롬프트 보조 도구가 또 다른 API 비용을 발생시키는 구조에 불만을 가진 사람

### 2.2 핵심 사용자 시나리오

> 김개발은 진행 중인 Next.js 프로젝트에서 알 수 없는 렌더링 에러를 만났다. 터미널에서 `promptcraft build`를 실행하면 도구가 프로젝트 구조(Next.js 14, TypeScript, TailwindCSS)를 자동으로 스캔하고, 분기형 질문을 통해 에러 메시지와 원하는 동작, 시도한 방법을 수집한다. 로컬에서만 동작하므로 이 과정에서 API 비용은 전혀 발생하지 않는다. 완성된 구조화 프롬프트가 클립보드에 복사되고, 이를 Claude에 붙여넣으니 한 번에 정확한 해결 코드를 받는다.

### 3.2 핵심 기능 상세

#### 3.2.1 프로젝트 로컬 스캔 엔진

`promptcraft scan` 실행 시 Node.js `fs` 모듈로 지정 디렉토리를 분석합니다. **외부 API 호출 없이 전량 로컬에서 처리됩니다.**

판별 요소:
- 언어 및 프레임워크 (`package.json`, `tsconfig.json`, `requirements.txt` 등 파싱)
- 주요 디렉토리 구조 (`/node_modules`, `/dist` 등 자동 제외)
- 환경 설정 파일 감지 (`.env` 존재 여부, 비밀 정보 필터링)

스캔 결과는 전체 파일 트리가 아닌 **요약본**으로만 프롬프트에 주입됩니다. 이것이 입력 토큰 절감의 1차 지점입니다.

#### 3.2.2 인터랙티브 분기 Q&A

사전 정의된 분기 트리 JSON을 기반으로 상황에 맞는 질문만 제공합니다. 상황 파악 자체에 LLM을 사용하지 않습니다.

```
[상황 선택]
├─ 에러/버그 해결
│   ├─ 문제 증상 (자유 입력)
│   ├─ 에러 로그 스니펫 (붙여넣기)
│   └─ 시도한 방법 (선택, 옵션)
├─ 신규 기능/컴포넌트 구현
│   ├─ 구현 목표
│   └─ 디자인/기술 제약 사항
├─ 코드 리뷰 및 리팩토링
│   └─ 집중 영역 (성능, 가독성, 보안 등)
└─ 개념/구조 설계 요청
    └─ 설계 목표 및 제약
```

#### 3.2.3 구조화 프롬프트 빌더

수집된 정보를 **섹션 분리 템플릿**에 조립합니다. 이 구조가 One-Shot 성공률의 핵심 메커니즘입니다.

**섹션 배치 원칙 (정보 우선순위 순):**

```markdown
## Role
[모델에게 부여할 역할 — 해석 없이 즉시 컨텍스트 고정]

## Stack & Environment
[스캔 결과 요약 — 불변 컨텍스트, 상단 배치]
- Framework: Next.js 14, TypeScript
- Styling: TailwindCSS
- Key paths: /src/components, /src/app

## Goal
[현재 달성하려는 목표 — 단일 문장으로 명확히]

## Current Situation
[Q&A로 수집된 현재 문제 상황]
버튼 컴포넌트 클릭 시 하이드레이션 에러 발생.

## Error / Evidence
[에러 로그 또는 재현 조건 — 스니펫만 포함]
(사용자 입력 에러 로그)

## Constraints
[응답에 반드시 반영되어야 할 제약 조건]
- 함수형 컴포넌트 + Hooks 유지
- 기존 Tailwind 클래스 수정 최소화
- 기존 파일 구조 변경 금지
```

**이 구조가 One-Shot을 이끄는 이유:**
모델이 각 섹션의 의미를 추론하지 않고 즉시 읽을 수 있어, 사고 토큰 전량이 문제 해결에 집중됩니다. 자연어 산문 방식 대비 동일 정보를 더 적은 토큰으로 더 높은 정확도로 전달합니다.

결과물은 Markdown 포맷으로 제공되며 자동으로 클립보드에 복사됩니다.

| 데이터 저장 | 로컬 SQLite (`~/.promptcraft/`) |
| 크로스 AI 호환 | 생성된 프롬프트가 Claude, GPT, Gemini 등 주요 모델에서 유효하게 동작해야 함 |

- 생성 내역 로컬 SQLite 저장

### 제외 (의도적 배제)
- **CLAUDE.md, GEMINI.md 등 전역 컨텍스트 파일 생성** — 매 AI 호출마다 대형 파일을 주입하는 방식은 토큰 낭비이며 핵심 목표에 반함
- **프롬프트 생성 단계의 LLM API 의존** — 이중 비용 구조를 야기하므로 MVP에서 완전 배제
- Python 기반 파서/스코어링 — Node.js 생태계로 통합
- 외부 LLM 연동 — 규칙 기반 100% 동작 확보 후 P2로 추가

## 11. 용어 정의

| 용어 | 설명 |
|---|---|
| 바이브 코딩 (Vibe Coding) | 코드 지식보다 AI 도구에 의도를 얼마나 잘 전달하느냐로 결과물을 만들어내는 개발 방식 |
| One-Shot | 단 한 번의 AI 호출로 추가 질문 없이 원하는 결과를 얻는 것 |
| 사고 토큰 (Thinking Tokens) | 모델이 응답을 생성하기 전 내부적으로 소비하는 추론 토큰. 구조화된 입력일수록 의도 파악에 낭비되는 사고 토큰이 줄어든다 |
| 섹션 분리 템플릿 | Role·Stack·Goal·Situation·Constraints를 명시적 섹션으로 분리한 구조화 프롬프트 포맷 |
| 분기 트리 | 상황별 최적 질문만 제공하기 위한 사전 정의 로직 트리 (JSON 기반) |
| 이중 비용 구조 | 프롬프트 생성에 LLM을 쓰고, 실제 작업에 또 LLM을 쓰는 기존 도구들의 구조적 문제 |
| 글로벌 컨텍스트 파일 | CLAUDE.md 등 프로젝트 최상단에 두고 매 AI 호출마다 주입되는 대형 Markdown 파일. PromptCraft는 이를 배제하고 문제별 최소 컨텍스트만 주입한다 |
