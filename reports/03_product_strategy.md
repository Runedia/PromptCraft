## 1. 사용자 여정(User Journey) 기반의 기능 최적화 제안

### 1.1 현재 설계된 사용자 여정 흐름

```
[설치] → [프로젝트 스캔] → [상황 유형 선택] → [분기 질의응답] → [프롬프트 생성] → [복사/저장]
```

### 1.2 단계별 마찰 구간 식별 및 개선안

#### 단계 1: 설치 (진입 장벽 — 심각도 높음)

| 마찰 요소 | 현재 상태 | 개선안 |
|---|---|---|
| Node.js 18+ 필수 | 별도 설치 필요 | 설치 가이드 + 버전 체크 스크립트 자동 실행 |
| Python 런타임 의존 | 스캔/스코어링에 별도 Python 필요 | Node.js 단일 스택으로 통합 검토 (fs + glob으로 대체 가능) |
| 전역 설치 명령어 | `npm install -g promptcraft` | 설치 후 자동으로 `promptcraft --help` 실행하여 성공 확인 유도 |
| 첫 실행 안내 부재 | 설치 후 다음 행동 불명확 | 설치 완료 시 "다음 단계: `promptcraft scan .`" 안내 출력 |

#### 단계 2: 프로젝트 스캔 (핵심 UX 결정 구간)

| 마찰 요소 | 현재 상태 | 개선안 |
|---|---|---|
| 스캔 결과 신뢰도 | 사용자가 결과 정확도를 확인할 방법 없음 | 스캔 완료 후 요약 출력 (감지된 언어, 프레임워크 목록) + 수정 옵션 제공 |
| 대규모 프로젝트 스캔 지연 | 5초 목표이나 모노레포 시 초과 가능 | 진행률 표시기(spinner) + 예상 시간 표시 |
| 경로 입력 방식 | 절대 경로 직접 입력 | 현재 디렉토리(`.`) 기본값 + 상대 경로 지원 |
| .env 존재 알림 | 내용 미수집 여부만 명시 | "보안: .env 파일 내용은 수집하지 않습니다" 명시 출력으로 신뢰 확보 |

#### 단계 3: 상황 유형 선택 (온보딩 핵심 구간)

| 마찰 요소 | 현재 상태 | 개선안 |
|---|---|---|
| 4가지 유형의 선택 기준 불명확 | 유형 이름만 제시 | 각 유형에 1줄 예시 추가 ("에러 해결 — 콘솔 에러 메시지가 있을 때") |
| 상황 미해당 시 대응 없음 | 4가지만 제공 | "해당 없음 / 기타" 옵션 + 자유 질문 모드 추가 (P2) |
| 첫 사용자의 유형 선택 불안 | 잘못 선택 시 처음부터 다시 해야 함 | 중간 뒤로가기 지원 + 유형 변경 옵션 |

#### 단계 4: 분기 질의응답 (핵심 가치 전달 구간)

| 마찰 요소 | 현재 상태 | 개선안 |
|---|---|---|
| 진행률 불투명 | API 응답에 `progress.estimated`는 있으나 CLI에 미표시 가능성 | "3/6 질문" 형식으로 CLI와 Web UI 모두 진행률 표시 필수화 |
| autoFill 값 수정 동선 | 스캔 결과로 자동 채움 후 수정 방법 불명확 | "Enter로 확인, 직접 입력으로 수정" 안내 문구 표준화 |
| 선택형 질문(multiselect) 학습 비용 | CLI에서 체크박스 조작법 비직관적 | 첫 multiselect 질문 등장 시 조작법 1회 안내 |
| 필수 항목 빈 입력 시 재질문 | 에러 메시지만 표시 | 재질문 시 이전 맥락과 힌트 함께 표시 |

#### 단계 5: 프롬프트 생성 및 활용 (가치 실현 구간)

| 마찰 요소 | 현재 상태 | 개선안 |
|---|---|---|
| 생성 결과 확인 경험 | 터미널 출력 or 파일 저장 | Web UI에서 복사 버튼 + 토큰 수 예측 표시 |
| 히스토리 재사용 동선 | P1 기능으로 분류 | MVP에 간이 히스토리(최근 3개) 포함 검토 — 재사용 습관 형성에 결정적 |
| 컨텍스트 파일 생성 연계 | 프롬프트 생성과 컨텍스트 파일 생성이 분리됨 | 프롬프트 생성 완료 후 "CLAUDE.md도 생성하시겠습니까?" 자연스러운 연계 유도 |

### 1.3 UX 최적화 우선순위 요약

| 순위 | 개선 항목 | 예상 효과 |
|---|---|---|
| 1 | 스캔 결과 요약 + 수정 옵션 | 사용자 신뢰도 및 정확도 대폭 향상 |
| 2 | 진행률 표시 (CLI + Web) | 중도 이탈률 감소 |
| 3 | 상황 유형 선택에 예시 추가 | 첫 사용자 온보딩 시간 단축 |
| 4 | 뒤로가기 지원 | 실수 복구 경험 개선 |
| 5 | 간이 히스토리 (최근 3개) MVP 포함 | 재사용 습관 형성 |

## 2. 서비스 가용성 확보를 위한 운영 정책 수립

### 2.1 로컬 설치형 특성 고려사항

PromptCraft는 클라우드 서비스가 아닌 로컬 설치형 도구이므로, 가용성 정책은 **설치 안정성**, **업데이트 관리**, **오류 복구** 중심으로 수립한다.

### 2.2 업데이트 정책

| 항목 | 정책 | 근거 |
|---|---|---|
| 버전 관리 방식 | Semantic Versioning (semver) | npm 생태계 표준, 사용자 예측 가능성 |
| 업데이트 알림 | 실행 시 최신 버전 감지 후 알림 (인터넷 연결 시) | `npm outdated` 체크 또는 GitHub Releases API |
| 업데이트 방법 | `npm update -g promptcraft` 단일 명령 | 사용자 자율 업데이트, 강제 업데이트 없음 |
| 분기 트리 업데이트 | 앱 업데이트와 독립적으로 `~/.promptcraft/trees/` 수동 배치 가능 | 커스텀 트리 손상 없이 내장 트리만 갱신 |
| DB 마이그레이션 | 앱 시작 시 자동 순차 마이그레이션 (`PRAGMA user_version`) | 기존 데이터 보존, 업그레이드 중 데이터 손실 방지 |
| 하위 호환성 | Major 버전 변경 시 마이그레이션 가이드 README 제공 | 사용자 히스토리/템플릿 데이터 보호 |

### 2.3 오류 복구 정책

| 오류 유형 | 감지 방법 | 복구 방안 |
|---|---|---|
| DB 파일 손상 | 시작 시 integrity_check 실행 | 자동 백업(`promptcraft.db.bak`) 복원 또는 초기화 안내 |
| 스캔 타임아웃 (5초 초과) | 타임아웃 감지 | 부분 결과 반환 + "대규모 프로젝트 감지, --depth 옵션 축소 권장" 안내 |
| Express 포트 충돌 (3000) | EADDRINUSE 에러 | 자동으로 다음 포트(3001, 3002) 시도 + 사용자에게 실제 URL 출력 |
| QnA 세션 만료 (30분) | SESSION_NOT_FOUND 반환 | 명확한 만료 안내 + 이전 답변 일부 복원 옵션 제공 |
| 클립보드 복사 실패 | clipboardy 예외 처리 | 복사 실패 시 프롬프트 파일로 자동 저장 + 경로 안내 |
| API Key 복호화 실패 | 복호화 예외 처리 | Key 초기화 후 재입력 안내 (데이터 유실 없이) |

### 2.4 데이터 안전 정책

| 항목 | 정책 |
|---|---|
| DB 위치 | `~/.promptcraft/promptcraft.db` — OS 사용자 공간 내, 타 사용자 접근 불가 |
| .env 내용 | 절대 수집/저장 금지. 파일 존재 여부(`hasEnv: true`)만 기록 |
| API Key 저장 | AES-256 또는 OS Keychain 활용 암호화 필수. 평문 저장 금지 |
| 스캔 캐시 무효화 | 디렉토리 파일 해시 기반 자동 감지. 변경 시 재스캔 |
| 사용자 데이터 초기화 | `promptcraft reset` 명령으로 DB + 설정 + 캐시 일괄 초기화 지원 |

### 2.5 사용자 지원 방안 (졸업과제 규모)

| 채널 | 내용 |
|---|---|
| README.md | 설치, 첫 사용, 주요 명령어, FAQ 포함 |
| GitHub Issues | 버그 리포트 및 기능 요청 수용 |
| `promptcraft --help` | 모든 명령어 인라인 도움말 |
| 에러 메시지 품질 | 모든 에러에 원인 + 해결 방법 포함 ("SCAN_PATH_NOT_FOUND: 경로가 존재하지 않습니다. 절대 경로로 다시 입력하세요.") |

## 3. 계획안의 약점 보완 로드맵 및 개선 우선순위

### 3.1 현재 계획의 핵심 약점 분석

| 약점 | 심각도 | 현재 계획 | 문제점 |
|---|---|---|---|
| **Python + Node.js 이중 런타임** | **높음** | 스캔/스코어링에 Python 사용 | 설치 복잡성 증가, 4인 팀 유지보수 부담 |
| **Web UI를 P1으로 분류** | **높음** | CLI MVP 이후 개발 | CLI에 익숙하지 않은 타겟 사용자 초기 채택 실패 위험 |
| **인터페이스 확정 의존성** | **중간** | API + DB + CLI는 Core 모듈 확정 후 시작 | 병렬 개발 불가로 일정 지연 위험 |
| **품질 스코어링 정의 부재** | 중간 | P2로 분류 | "좋은 프롬프트"의 기준이 없어 핵심 가치 측정 불가 |
| **컨텍스트 파일 포맷 변경 대응** | 중간 | 정적 템플릿 | AI 도구사의 포맷 변경 시 수동 업데이트 필요 |
| **Windows 전용 초기 출시** | 낮음 | Windows Node.js 18+ 명시 | 오픈소스 기여자/맥 사용자 제외 |

### 3.2 보완된 로드맵 제안

#### 수정 전 계획 (원안)

| 월 | 마일스톤 | 주요 작업 |
|---|---|---|
| 1개월차 | 설계 완료 | 요구사항, 분기 트리 JSON, DB 스키마 |
| 2개월차 | 핵심 기능 | 스캔, 질의응답, 프롬프트 생성, CLI |
| 3개월차 | 통합 + 2차 기능 | Web UI, 히스토리, 템플릿, 컨텍스트 파일 |
| 4개월차 | 완성 + 발표 | 버그 수정, 스코어링, AI 연결, 데모 |

| 월 | 마일스톤 | 주요 변경 사항 | 이유 |

## 4. 종합 의견

PromptCraft의 제품 전략에서 가장 시급한 과제는 **"초기 사용 성공 경험"의 설계**다. 아무리 기능이 충실해도 첫 사용에서 마찰이 크면 채택되지 않는다.

**핵심 3대 권고사항:**

1. **Python 의존성 제거**: 설치 경험을 `npm install -g promptcraft` 한 줄로 완결지어야 한다. Node.js `glob`, `fast-glob` 라이브러리로 파일 스캔을 완전히 대체할 수 있다.

2. **Web UI 조기 착수**: P1이 아닌 P0 수준으로 실질적으로 상향하여 CLI 미숙자도 첫날부터 사용 가능한 환경을 제공해야 한다.

3. **인터페이스 먼저, 구현 나중**: Core 모듈 인터페이스(ScanResult, QnA State 스키마)를 1주 내 확정하면 4인이 독립적으로 병렬 개발 가능하며, 전체 일정을 안전하게 단축할 수 있다.
