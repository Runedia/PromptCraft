### 1.3 차별점 분석

**PromptCraft의 핵심 차별점:**

1. **도구 독립성 (Vendor-Agnostic)**: 경쟁 서비스 모두 자사 AI 도구에 종속. PromptCraft는 단일 프로젝트 스캔으로 Claude, Gemini, Cursor 등 복수 포맷을 동시 출력.

2. **프로젝트 컨텍스트 자동화**: 기존 서비스는 개발자가 CLAUDE.md, .cursorrules 등을 직접 작성해야 함. PromptCraft는 코드베이스를 자동 분석하여 생성.

3. **분기 질의응답 기반 구조화**: 단순 텍스트 입력이 아닌, 상황별(에러 해결/기능 구현/코드 리뷰/개념 이해) 안내 질문으로 초보자도 고품질 프롬프트 작성 가능.

4. **완전한 오프라인 동작**: API Key 없이도 모든 핵심 기능 동작. 인터넷 제한 환경(사내망, 보안 구역) 활용 가능.

**차별점 부재 확인 (리스크 항목):**

| 항목 | 평가 | 근거 |
|---|---|---|
| 프롬프트 품질 향상 효과 | **차별점 약함** | AI 도구 자체의 대화 기능(Few-shot, RAG)으로 대체 가능 |
| 커스텀 트리 확장성 | 차별점 있음 | 타 서비스에 없는 JSON 기반 커스텀 분기 구조 |
| Web UI 편의성 | **차별점 약함** | ChatGPT, Claude 등 이미 직관적 Web UI 보유 |
| 히스토리/템플릿 관리 | **차별점 약함** | Notion, 메모앱 등 일반 도구로 대체 가능 |

## 2. 목표 시장 점유율 및 비즈니스 모델 타당성 검토

### 2.1 타겟 시장 규모 추정

| 세그먼트 | 규모 추정 | 근거 |
|---|---|---|
| 국내 AI 코딩 도구 활성 사용자 | 약 50~100만 명 | GitHub Copilot 국내 사용자, 개발자 인구 통계 |
| 학생~주니어 개발자 비중 | 약 30~40% | 대학 CS 전공자 + 부트캠프 수료생 |
| 실질 타겟 (오프라인 도구 선호) | 약 5~15만 명 | 보안/비용 이유로 로컬 도구 선호 세그먼트 |

### 2.2 비즈니스 모델 타당성 (졸업과제 맥락 기준)

**졸업과제 특성상 수익 모델은 평가 외 항목이나, 향후 상용화 가능성 관점에서 검토:**

| 모델 | 적합성 | 한계 |
|---|---|---|
| 오픈소스 무료 배포 | **높음** | 직접 수익 없음, 포트폴리오/커뮤니티 가치 |
| npm 무료 패키지 | **높음** | 유사 도구와의 경쟁에서 채택률 확보 유리 |
| 프리미엄(AI 기능 유료화) | 보통 | P2 기능(AI 연결) 완성도 확보 후 가능 |
| SaaS 전환 (클라우드 호스팅) | **낮음** | 로컬 설치형 정체성과 상충, 개발 부담 과중 |
| 기업 라이선스 | 낮음 | 졸업과제 규모로 기업 영업 불가 |

**타당성 결론**: 졸업과제로서 **오픈소스 npm 패키지**로 배포하는 모델이 가장 현실적. 학술적 성취(논문, 발표)와 실용적 결과물(GitHub 스타, 실사용자)을 동시에 달성 가능.

### 2.3 시장 진입 시 주요 위협

| 위협 | 심각도 | 설명 |
|---|---|---|
| 주요 AI 도구의 기능 내재화 | **높음** | Claude Code, Cursor 등이 자동 컨텍스트 생성 기능을 추가하면 PromptCraft의 차별점 소멸 |
| 사용자 관성 | **중간** | 개발자들은 이미 익숙한 도구(ChatGPT 직접 사용)를 바꾸기 어려워함 |
| 프롬프트 엔지니어링 필요성 감소 | **중간** | AI 모델 자체의 추론 능력 향상으로 구조화 프롬프트 필요성 감소 추세 |

## 4. 종합 의견

PromptCraft는 **시장에 명확한 빈틈(niche)을 공략하는 도구**다. 특정 AI 도구에 종속되지 않는 범용 컨텍스트 파일 생성기라는 포지션은 현재 경쟁 서비스 중 유사한 제품이 없어 단기적으로 차별화된다.

그러나 **장기적 위협은 주요 AI 도구의 기능 흡수**다. Claude Code, Cursor 등이 자동 프로젝트 분석 및 컨텍스트 생성 기능을 내재화하면 PromptCraft의 핵심 가치가 희석된다. 따라서 졸업과제 완성과 오픈소스 커뮤니티 확보를 단기 목표로 설정하고, 상용화는 시장 반응 이후 판단하는 것이 현실적이다.

**가장 시급한 개선 권고사항**: Python 런타임 의존성 제거를 검토하여 설치 장벽을 최소화하고, Web UI를 P0 수준으로 상향하여 비CLI 사용자층을 초기부터 포용해야 한다.
